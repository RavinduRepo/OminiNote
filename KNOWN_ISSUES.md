# KNOWN_ISSUES.md

Tracks issues and cross-platform gaps found during codebase audits. Keep this in sync with reality: when an item is fixed, remove it (or move it to "Fixed") instead of leaving it marked done; when new issues are found, add them here rather than only mentioning them in conversation.

## Open issues

### Canvas v1 limitations (deliberate scope cuts — see CANVAS_SPEC.md §17/§19)

- **Selection is single-page**; can't lasso across page boundaries.
- **Undo/redo history is in-memory** (~100 ops), cleared when the canvas closes.
- **S-Pen button-to-erase works while the pen is touching** (`kPrimaryStylusButton` / inverted stylus); a pure hover-button tool toggle isn't implemented.

### Cloud sync (Drive, v2)


- **Setup prerequisites (not code — Google Cloud config).** Sync needs OAuth clients in the Cloud project: a **Desktop** client (id/secret in `.dart_defines.json`) and an **Android** client registered for package `io.github.ravinduRepo.omininote` + the signing SHA-1 (debug SHA-1 per dev machine for `flutter run`, release SHA-1 `34:48:37:E1:...` for store builds). Each dev machine needs its own Android debug OAuth client registered with its local debug keystore SHA-1. The consent screen is now **In production** with `drive.file` classified as non-sensitive — no user cap, any Gmail can sign in. Branding verification is pending (blocked on `github.io` shared-domain limitation; cosmetic only — app works without it).
- **Open-canvas live merge is now wired** (07/07/26 — see Fixed). Remaining window: two devices editing the *same page* within one poll cycle merge via union on the next round trip — ink is never lost, but LWW-resolved fields (an element's position, page background) can briefly flip-flop before settling.
- **Structural *membership* is now union-merged; only folder grouping + ordering stay LWW (07/18/26 — see Fixed).** A canvas/page/bookmark/attachment added concurrently on two devices is no longer dropped — `section.json` unions its canvas leaves and `canvas.json` unions its page rows + bookmarks + attachments. What remains LWW on `section.json`/`canvas.json`: **the arrangement** (order, and super-section/folder grouping). So if two devices reorder the same tree offline, one order wins; and a brand-new *folder* created on the losing side loses its grouping (its canvases still survive, appended at the top level). The losing side's new items land at the **end**, not their original slot. Fractional indexing (plan §1.5) would additionally preserve order/position; folder-node union would preserve loser-side grouping — both deferred as lower-value than the membership loss, which is fixed.
- **No Conflicts Inbox / conflicted copies yet** (plan §3.4). A delete that races an edit resolves by the `(rev,updatedAt,deviceId)` tuple rather than restoring + prompting.
- **A purge permanently discards unsynced offline edits to the purged item.** Inherent to "completely wipe": a device that edited a notebook offline and reconnects after the notebook was purged has its uploads for that subtree dropped by the push filter — the edits are gone (the alternative would repopulate Drive with orphaned content). The 30-day bin window before auto-purge is the mitigation.
- **`appDataFolder` not used.** The Drive tree lives in a **visible** `omininote/` folder in My Drive (nice as a user-inspectable backup) under the `drive.file` scope, rather than the hidden `appDataFolder` the plan suggested.

### Multi-account sync (Phase 2 — `SYNC_SHARING_PLAN.md`)

- **Moving a notebook between accounts loses edits made on a device that hasn't synced at move time.** A move fully syncs the notebook DOWN from its source account first (so cloud+local data is preserved) and re-keys it to a new id, but a *third* device that edited the notebook and hasn't uploaded yet will, when it does, write to the now-tombstoned old id → those edits are orphaned. Advise: sync all devices before moving a notebook to another account. (Inherent to any move in an eventually-consistent system.)
- **A moved notebook's old-account content lingers on the old Drive for up to 30 days.** The re-key tombstones the old notebook id (so other devices drop it); when that tombstone expires, the GC's purge pass now deletes the old Drive folder too — but only from a device signed into the *old* account at the time. If no device ever signs into that account again, its orphaned folder stays.
- **Moving requires being signed into the notebook's current account on the device doing the move** (to sync it down first). If you aren't, the move is refused rather than risk data loss.
- **Removing an account is per-account but warns on *global* unsynced changes.** The remove-account dialog's "unsynced changes" warning and the gate on the purge checkbox use the whole-journal pending flag, not a per-account count (the dirty journal isn't account-tagged) — so pending edits on account B can block purging account A's local copies. Conservative, not wrong.

### Clipboard / platform

- **Upstream: `super_native_extensions` crashes the process on an unreadable clipboard URI — guarded, not cured (07/17/26).** `ClipDataHelper.getFormats` calls `ContentResolver.getStreamTypes()` on any clipboard item carrying a URI and doesn't catch `SecurityException`; when the item points at a non-exported `FileProvider` we hold no URI grant for, Android throws on the **main Looper inside the plugin's Java** → `FATAL EXCEPTION: main ... Permission Denial: opening provider androidx.core.content.FileProvider` → process death. **Not catchable from Dart** (it never becomes a Dart exception). Still unfixed on their `main`; 0.9.1 is the newest release (13 months old), so there is nothing to upgrade to — worth filing upstream (a one-line try/catch; their own `coerceToPlainText` already does it). **Everyday trigger:** a desktop→phone clipboard sync (Samsung Link to Windows and friends) parks such a URI on the clipboard — this reads as "only crashes while the laptop is connected", but the laptop is the *source of the clipboard content*, not the cause; copying from a phone app that does the same crashes with no cable attached. **Mitigated** by `clipboardIsReadable()` + `MainActivity.canEnumerateClipboard` (Kotlin, channel `omninote/clipboard_guard`): we make the same call first where we can catch it, and skip `super_clipboard` on a throw (costing nothing — a throw means the data was unreadable anyway; paste falls back to plain text). Consequence: pasting an image from such a source **silently does nothing** instead of crashing. Fails open everywhere else. **Any new `SystemClipboard.read()` call site must go through the guard** — the format enumeration is inside `read()` itself, so every read is exposed.
- **Benign `EPIPE (Broken pipe)` warnings on Samsung when copying an image** — logged by `super_native_extensions`'s `DataProvider$PipeDataWriter` (observed live on an SM X510, 07/07/26). One UI's clipboard manager opens the offered image stream repeatedly to build its preview thumbnail and closes the pipe early each time; the plugin catches it and the actual paste target re-requests and receives the full data. Warning noise only — do not chase it as a copy failure.
- **Image clipboard now uses `super_clipboard`** (07/07/26; replaced `pasteboard`) — copy *and* paste of images works on Android, Windows, Linux, macOS, and iOS (the latter two are future targets but already covered). Remaining caveats: on Android both directions depend on the other app using the standard clipboard image contract (all mainstream apps do); Linux still needs a clipboard manager present on very bare setups. Where unsupported, copy/paste falls back to the app-internal clipboard (full fidelity in-app) and OS text. Note `minSdk` is now 23 and first build after `flutter clean` downloads the plugin's precompiled native core (slow once, not broken).

### Connections (internal links — Phases 1–3 shipped 07/19/26)

- **Markers and connections are coupled (Model A, 07/22/26):** an on-canvas link marker *is* the connection's visual body. Deleting a **standalone** marker element (lasso-select + delete) breaks its connection and removes the reciprocal marker on the other side (`LinkService.removeLinkForMarker`); removing a connection from the Connections sheet ✕ deletes both markers (`removeLinkAndMarkers`). Only standalone markers (a text element that is just one hyperlink) are touched — an inline `[[` link inside a larger text box is left alone (`standaloneMarkerUri`). For a durable clickable jump that ISN'T a tracked connection, use a plain/external URL. Open-canvas-first dispatch mirrors marker insertion; the reciprocal removal on the open canvas rides `removeElement` (undoable, symmetric with insertion). **Undo caveat:** undoing a marker delete restores the marker(s) but the tombstoned record stays dead (same class as the old paste-undo gap) — re-link if needed.
- **Undoing a pasted/inserted link leaves its registry record** (the connection row survives, resolving at page level) — this is the *paste* undo path; deletion now breaks the connection (see Model A above). Same page-granularity resolution for deleting a linked element/word later.
- **Moving a linked element between pages auto-corrects its links (07/22/26):** a cross-page move within a canvas (`_reparentSelection`) updates every reference to the moved elements to their new page — the connection **records** in `links.json` (`LinkService.remapElementsToPage`, store-wide), the on-canvas link-run **URIs** on the moved canvas (`_remapLinkUrisOnCanvas`), **and — cross-canvas — the marker on the OTHER canvas that points at the item**, closed or open-in-a-split: the record pinpoints that marker's page, so `remapElementsToPage` dispatches a targeted URI rewrite there (open canvas → `onRemapMarkerUris` in memory; else `NotebookService.remapLinkMarkerUrisOnPage` on disk — the same open-first/disk pattern Model A uses to *delete* a far marker). All pure via `remapLinkUriPage`. Runs from the move op's apply **and** revert (undo/redo carry the links back). So both the Connections menu and every marker tap follow the item to its new page, in any canvas. (Edge: if you move only the linked content and leave its reciprocal marker physically behind, the endpoint's page follows the content — the stranded marker's own file entry is a minor cosmetic mismatch, navigation is correct.)
- **Word-level links are sources, not targets** (deliberate simplification, user-accepted): navigating *to* a linked element highlights the whole element, and a text box's backlinks are element-level. Word-range *targets* would break as text is edited.
- **The `[[` picker commits the text session** before opening (a modal sheet's taps would kill the session anyway); after picking, the link is spliced into the committed box and the editor is closed — tap the box again to continue typing.
- **In-text linking — search OR paste, caret OR selection (07/22/26):** the `[[` trigger **and** a new **link button in the text style row** both open the target picker, which now has a **Paste** button — a copied internal/external link becomes the target directly (create the connection), copied plain text seeds the search. No selection → the target's name is inserted as a link at the caret (`insertLinkAtCaret`); a **selected span** is hyperlinked in place, keeping the selected text as the clickable display (`applyLinkToRange`/`setLinkOnRuns`) — e.g. link one line of a to-do list. Works on mobile + desktop (the button rides `buildTextStyleRow`, shown by both the normal bottom bar and the full-screen panel; routed via the screen hook `CanvasController.addTextLinkHook`). Registers via `addLinkWithReciprocalMarker`, so it shows in Connections and a pasted element-link drops a reciprocal marker. **Limitation (Model A gap for INLINE links):** an in-text link lives inside a box with other content, so it's not a *standalone* marker — deleting the box or removing the connection from the sheet leaves the inline hyperlink as possibly-dangling styled text (the run's URI still navigates; the record may be gone). De-link it with the ✎ sheet. Full inline coupling (auto de-link a run on connection removal) is deferred.
- **Editing a link's display text mid-run via typing splits the link styling:** typed characters never extend a hyperlink (adoption clears `link` from the typing style), so use the ✎ sheet to change a link's text cleanly.
- ~~A lasso selection linked from the *other* side gets no on-canvas marker.~~ **Fixed (design round 2, 07/20/26):** linking to an element endpoint from anywhere (canvas paste, in-editor paste, Connections sheet, ✎ retarget fallback) drops a **reciprocal marker** next to the target elements — `LinkService.addLinkWithReciprocalMarker`, open-canvas-first dispatch. Repeated markers at the same anchor **stack into a vertical list** (`stackBelowObstacles`) instead of overlapping. Remaining gap: undoing the source-side *paste* doesn't remove the record or the other canvas's reciprocal marker (cross-canvas undo can't reach it) — but **deleting** the marker or removing the connection from the sheet now cascades to both sides (Model A, above).
- **Design rounds 1–2 (07/20/26) user-verified on device:** mobile one-level-up landing + home-card glow, lasso link markers, per-row ✎ edit, external-URL targets, scroll-to-glow in long lists, reciprocal markers + stacking all confirmed working.
- **`links.json` uploads whole to every connected account.** Records are tiny id tuples plus name snapshots + optional labels, but those names/labels are visible to any account the device syncs — acceptable for a personal multi-account setup, worth revisiting if sharing ever spans users. An endpoint whose notebook lives on another account (or is local-only on another device) simply resolves as dead there.
- **Element endpoints resolve at page granularity.** A link to a lasso selection stays "alive" as long as its page row exists, even if the specific elements were later deleted (checking would need a page-file read per row); the landing view shows the truth. Phase 2's landing glow will read the page anyway and can tighten this.
- **A restore's revival and cross-device caveats mirror the bin's:** dead links revive automatically when the target is restored (aliveness is checked at display time), but a link created against a purged item stays dead forever (the marker never leaves).
- **Not yet device-verified** — built + unit-tested (`test/link_model_test.dart`); exercise the copy→paste flow, both shells' navigation, and a two-device sync round before relying on it.

### Notebook sharing (`.omninote` bundles — Phase 3)

- **Samsung "My Files" won't direct-open a `.omninote` file** (verified on an SM X510). Because `.omninote` is a custom, unregistered extension, Samsung's file manager shows "No app can open this" / a Play-Store search *regardless* of the app's `VIEW` intent-filters — it checks against known types before resolving intents. No manifest change fixes it (`application/octet-stream`, `application/zip`, `application/*`, and a path-pattern filter were all tried live). **Workaround: "Share → Omininote" from My Files** (the `SEND` path works everywhere). Direct-tap ("Open with") *does* work from Telegram, Google Files, and most other apps, which send a matchable MIME type. The app deliberately matches `application/zip` + `application/octet-stream` (not `*/*`) to avoid appearing in "Open with" for images/video/etc.
- **`omninote://` share links aren't tappable in messaging/email apps.** Custom URI schemes are only *auto-linkified* by browsers for `http(s)://` — WhatsApp/Telegram/Gmail/etc. render `omninote://import?id=…` as plain, non-tappable text; the recipient has to paste it into a browser address bar to invoke it (which then opens the app and imports fine). The mechanism (host public bundle → download → import) works end to end; the shareability is the limitation. A universally tappable link needs an `https://` **App Link / Universal Link** backed by a hosted domain (a `.well-known/assetlinks.json` + redirect) — deliberately out of scope (no backend). Kept as-is by user choice; "Send a copy" (file / share sheet) is the reliable path.

### Robustness

- **Stuck stylus-hover clears on next finger touch, not instantly on pen-lift (07/19/26).** A pen hovering a button on mobile shows a hover shadow/tooltip but emits **no** `PointerExitEvent` on lift, so the last-hovered widget stayed hovered. A pure hover→lift produces no event to react to, so `main.dart`'s `_onRootPointer` (a mobile-only root `Listener`) instead remembers the hovering stylus's device id and, on the **next non-stylus** press/release (a finger tap — exactly when the stale shadow is in the way), dispatches a synthetic off-screen `PointerHoverEvent` for that device to fire the missing exit + dismisses tooltips. Consequence: the stale shadow can linger between pen-removal and the next touch (Flutter gives no signal for "pen left range", so instant clearing isn't possible without a platform hover-exit). Best-effort + try/caught; desktop mouse hover untouched. An earlier version keyed on the pen's *own* pointer-up and never fired for a touchless hover.
- **IDs are timestamp-based** (`millis + 3-digit sequence` / `micros + sequence`) — fine in practice, but not UUIDs.
- **Syncfusion licensing**: export builds on `syncfusion_flutter_pdf` under the Community License (user-approved 07/06/26). If the project outgrows eligibility, swap via the `PdfExporter` interface.

### Distribution / installation

- **macOS release build is unsigned.** Distributed `.app` bundles trigger Gatekeeper ("Apple could not verify…") because there is no Developer ID certificate. Users must run `xattr -cr OminiNote.app` or use System Settings → Privacy & Security → Open Anyway on first launch. Resolving requires an Apple Developer account ($99/yr) for Developer ID signing. Mac App Store distribution requires a separate Paid distribution certificate.

- **Android: debug and release builds install as separate apps** if the debug build was ever installed before the package rename (old `com.example.omininote` stays alongside new `io.github.ravinduRepo.omininote`). Users upgrading from a pre-rename debug build must manually uninstall the old one.

### Features — planned

- **Android multi-window: a non-owner window's edits don't sync until it becomes the sync owner** (07/18/26 — foundation shipped, see Fixed "true Android multi-instance"). Windows + the single-sync-owner lease are in place (no corruption). The remaining gap: only the **owner** window journals + pushes; a *secondary* window edits local files fine, but those edits stay local-only until that window becomes the owner (the other closes) **and the file is touched again** (a non-owner can't journal without a second writer on the shared journal). The proper fix is **per-instance journals** (`sync_journal_<instanceId>.json`) that the owner drains — deliberately deferred (needs Android device/DeX testing, which wasn't available when it was built). Also: **editing the same canvas in two windows is last-writer-wins** on that page file (same class as two devices; the Settings note warns to open different canvases). And the whole feature is **not yet device-verified** — the manifest/launch/lease logic is unit-tested + analyzer-clean but never run on real Android multi-window.

- **Linked copies of sections/canvases.** When duplicating a section, super-section, or canvas, offer a "Linked copy" checkbox: changes in either copy propagate to the other (like OneNote's page sync). Architecture TBD — simplest approach is a `linkId` on the section/canvas level mirroring the existing `TextElement.linkId` pattern.

- **Note summarisation.** Feed canvas text content to an LLM API and insert a summary text element. Depends on voice recording (for audio transcription path) or can start with text-only canvases.

  - **Not device-verified yet.** Implemented + unit-tested + analyzer-clean, but recording/playback needs a real mic/device to confirm end to end (no mic in the build session). Test on the tablet: record → draw → stop → play back → audio-sync glow.

Deliberate — it needs no per-stroke schema change.

- **Read-aloud / text-to-speech — remaining caveats** (feature shipped 07/19/26, see Fixed; extraction layer is future-proof for image/handwriting OCR):
  - **Device status: Android confirmed** (read-aloud + highlight + tap-to-jump). **Windows** initially crashed on play — the flutter_tts Windows plugin was re-entered from its own completion callback; fixed by deferring the next utterance to a fresh event-loop turn (`Future(() => _speakCurrent())`) + `_gen` guarding + dropping `awaitSpeakCompletion(true)` — **re-verify on Windows.** macOS/iOS/Linux still unverified. (Note: a stray `xyz.luan/audioplayers … non-platform thread` line in the Windows log is a lingering-recording-player red herring, not the TTS crash — recording playback itself works on Windows.)
  - **Pause resumes at the start of the current sentence, not the exact word** (stop + re-speak-the-sentence is the only cross-platform-robust pause). **Speed caps at the engine max** (`0.5×mult` clamped to ≤1.0), so 2.0× won't sound twice as fast, and the perceived rate differs per platform.
  - **Linux needs `espeak` or `spd-say` installed** — `flutter_tts` has no Linux plugin, so `TtsService` shells out. Without either, `isSupported` is false and the reader toasts "install espeak or spd-say". (espeak/spd-say rate/voice quality is basic vs the native engines.)
  - **Windows build prerequisite: `nuget.exe` must be on PATH.** flutter_tts's Windows plugin CMake calls NuGet to fetch the WinRT projection; without it the build fails with `nuget.exe not found`. Fix once: `winget install Microsoft.NuGet` (or drop `nuget.exe` from dist.nuget.org on PATH), reopen the terminal, then `flutter clean` (the failed CMake configure is cached) and rebuild. Not needed on Android/macOS/Linux.
  - **Scanned/image-only PDFs have no text layer**, so `PdfTextExtractor` returns nothing for them — that's exactly the gap the planned **image-OCR `PageTextSource`** fills (drop-in: implement `PageTextSource`, add to `CanvasController._textSources`). Same for handwritten ink (planned handwriting-OCR source).
  - **Read-along highlight + tap-to-jump now work for BOTH typed and PDF text** (07/19/26): typed is per-sentence (exact glyph boxes); PDF is per-line (syncfusion `extractTextLines` bounds, scaled to the normalized page, mapped per sentence). Tap-to-jump fires only in non-inking tools (inking taps draw). Caveat: PDF line bounds assume syncfusion's top-left origin + that the extracted page size matches the pdfrx size used at import (true for normal pages; a rotated/odd-cropbox PDF could shift the highlight — verify on device).

  - ~~Manually pinning a voice silently forced it onto every language, so e.g. Sinhala text stopped auto-selecting a Sinhala voice~~ (fixed 07/19/26) — see the "Non-Latin auto-switch" note under Voice selection in `CLAUDE.md`.
  - **Reading voice + reading position are device-local, not synced** (by design today, like `SettingsService`'s other per-device prefs — viewport, eraser size, promoted toolbar): considered syncing them (raised 07/19/26) but voice *names* aren't portable across OSes (a Windows "Microsoft Zira" pick means nothing on Android/Linux, so shipping the raw name+locale would just silently fail to apply on another device) and settings.json is deliberately outside the sync surface (`isSyncedRelPath` only covers `notebooks.json`/`notebooks/…`) — adding a new synced-preference channel is real new plumbing (merge rule, echo suppression), not a small change. ## Fixed

- **"Sync push error: Out of Memory" + periodic freezes while writing (07/24/26).** Asset uploads did `file.readAsBytes()` (whole file in heap, on the main isolate) and googleapis' default **multipart** uploader then built the body **base64-encoded** (verified in `_discoveryapis_commons` — `Content-Transfer-Encoding: base64`), so a 100MB+ PDF/video cost ~2.5–3× its size in peak Dart heap plus seconds of main-isolate CPU → frame skips, then `OutOfMemoryError`; the failed rel never left the dirty journal (5 full re-reads per drain, the throw also starved rels queued behind it), so every autosave re-triggered the giant upload — the "can't continuously write" freeze loop, reproduced on-device with a 119MB PDF asset. Fixed transport-only (sync semantics untouched): assets now upload via `DriveService.uploadBinaryFile` (Drive **resumable** protocol, streamed from disk in 1MB chunks; the already-on-Drive skip runs **before** any bytes are read, and a retry re-opens the file instead of re-reading it) and download via `downloadToFile` (streamed to `.tmp` + rename); the share-link bundle upload (`uploadSharedBundle`) streams from its temp ZIP the same way.
- **Null-`syncTarget` notebooks silently rebound to whichever account a device added first (07/24/26 — the cause of the orphaned-notebooks incident above).** A null target means "the default account", and the default is install-order-dependent. Two-point fix, no rev bumps: the per-account index pull (`MergeEngine.mergeNotebooksIndexScoped(stampSyncTarget:)`) stamps every reconciled null-target entry with the account it was pulled from (deterministic across devices — a notebook id lives on exactly one account's Drive — so it converges byte-identically), and the per-account index upload (`syncedIndexJsonFor`) stamps null targets in the uploaded copy (the filter just decided the entry belongs to that account; writing it explicitly stops a different-default device from rebinding on pull). Explicit targets are never restamped; foreign/local-only entries untouched. Covered in `test/merge_engine_test.dart` (stampSyncTarget group). **Incident outcome (same day):** a mid-incident device snapshot showed 24 of 26 notebook ids present on the account's Drive but missing from the local index — a clear-data + fresh re-signin to the owning account restored everything (its Drive-side `notebooks.json` had survived intact, likely because the concurrent OOM bug was blocking the push drain); no Drive-revision recovery was needed. The 9 surviving pre-multi-account notebooks still carried null targets and are stamped on the first sync round with this fix.

- ~~Linux (Arch/Hyprland/Omarchy): the signed-in Google account was lost on every app restart — you had to re-login each launch~~ (07/20/26) — root cause was **not** the app losing the token: `flutter_secure_storage` stores the refresh-token blob in the gnome-keyring Secret Service, but on this setup the keyring is named `Default_keyring` (not the special `login` keyring), and the daemon is started via **systemd socket activation** — so PAM never unlocks it with the login password and it boots **LOCKED** every session, with no gcr prompter under Hyprland to unlock it on demand. `AuthService._load()`'s `_secure.read()` therefore returned nothing at startup and the session looked signed-out, even though the token was safely on disk in the (locked) keyring. Fix (`lib/services/auth_service.dart`): a **Linux-only file backstop** — secure storage stays the **primary** store on every platform (Windows DPAPI / macOS Keychain / Android Keystore are reliable and completely untouched), but on Linux the accounts blob is mirrored to a private `0600` `~/.local/share/…/.auth_accounts.json`. `_load` reads secure storage first and only falls back to the file when the keyring yields nothing/throws; a *successful* keyring read also re-seeds the file, so an already-signed-in user's token gets backed up on the next launch with **no forced re-login**. `_persist` writes both (keyring write is now best-effort so a locked keyring can't throw up the stack) and deletes the file on sign-out. Analyzer-clean; the token file is standard desktop practice (gcloud/rclone do the same) but scoped to `drive.file`.
- ~~No true OS multi-window (separate app windows) on Android~~ (07/18/26 — **launch device-verified on a Galaxy Tab S9 FE**; sync-across-windows not yet) — **open separate OminiNote windows.** Manifest `android:resizeableActivity="true"` + **`launchMode="standard"`** (was `singleTop`, which made the launch `START_DELIVERED_TO_TOP` — reusing the current window instead of a new instance; device-diagnosed via `ActivityTaskManager` logs, now `LAUNCH_MULTIPLE`/`result code=0`). `MainActivity`'s `omninote/multiwindow` channel launches another `MainActivity` with `NEW_TASK|MULTIPLE_TASK` + a unique per-launch `action` + `ActivityOptions.setLaunchBounds` (cascaded ~60% floating window on DeX/freeform), **deferred to the next frame** (`decorView.post{}` — a synchronous launch inside the tap dispatch was dropped in a floating window: the reason a plain tap did nothing while an awkward multi-touch release worked). Surfaced via `MultiWindowService` + a Settings → Multi-window → "Open new window" tile (Android N+ only). Trade-off of `standard`: a warm open-with of a file may now open in a new instance (the launcher itself still reuses the running task — no MULTIPLE_TASK there). **Each extra window runs in its OWN process** (`MainActivityW1/W2/W3`, `android:process=":wN"`, chosen by `openNewWindow` from the not-currently-running slots) — device-required, because opening a **PDF** canvas with two windows in one process aborted the app (`SIGABRT`, *"Cannot invoke native callback from a different isolate"* — pdfrx/pdfium's FFI callbacks are isolate-bound). Separate processes = isolated native state = no clash (device-verified: two windows, both PDF canvases, no crash). Caps at main + 3 windows. **Opening the same canvas in two windows is deliberately ALLOWED** (no lock — a same-canvas handoff was prototyped then removed at the user's call): concurrent edits to *different pages* merge via the per-page set-union sync; two windows editing the *same page* is a local last-writer-wins on that page file (an accepted edge, matching the two-device model). **Safety (the load-bearing part):** each window is its own `FlutterEngine`/isolate with its own sync/storage singletons over one filesystem, so `SyncCoordinator` elects a **single sync owner** via a heartbeat lease file (`sync_owner.lock` at appDir root, not synced): the owner runs GC + the Drive poll/push loop + writes the shared `drive_index.json`/`sync_journal.json`; non-owner windows skip all of it (`SyncService._active` gates `_markDirty`, so a non-owner never journals/pushes → no second writer → no corruption). A lone window always wins the lease instantly → **single-instance behavior is unchanged**; on owner close another window takes over (stale-lease detection, `leaseHeldByOther` unit-tested). Remaining gaps tracked under "planned" (non-owner edits sync only after takeover; same-canvas-in-two-windows LWW; device verification).

- ~~Importing a large `.omninote` bundle could crash on Android (OOM)~~ (07/15/26) — **streaming import** (#4b, symmetric to the 07/14 streaming export). Import used to inflate the whole zip into a `Map<String,Uint8List>` in an isolate and copy it back to main (peak ≈ decompressed tree ×2), on top of the picker/http body already holding the input in RAM. Covered by `test/notebook_bundle_import_test.dart` (lossless round-trip, invalid/corrupt rejection writes nothing, zip-slip guard).

- ~~Move an item to another page and back deleted it; and undo/fresh-id revival broke the undo chain~~ (07/12/26) — **rev-based (LWW) deletion for strokes & objects**, unifying them with how the Recycle bin already deletes notebooks/sections/canvases. Root cause of both: stroke/object deletion was a *grow-only tombstone set* (a tombstone could be added but never cancelled), so any "reverse a deletion" (undo, move-back) was unsafe across sync. `MergeEngine.mergePage`'s filter is now `element alive iff no tombstone || element.rev > tombstone.rev`; a tombstone records the element's rev at delete time (`EraseTombstone.rev`, previously always default `1`). Delete = tombstone at `el.rev`; undo/revive/move-back = keep the SAME id and bump the rev above the tombstone (`_bumpAliveOn`), never un-tombstoning. This fixes (1) **move A→B→A** — each move bumps rev, so the returning element out-revs the stale origin tombstone; (2) **the undo chain** — same-id revival keeps earlier ops able to find the element (the prior fresh-id fix orphaned it); and (3) a latent variant of the partial-eraser-undo-across-sync bug in split-paste / `deleteLinkedText` (they used to un-tombstone on undo). All delete paths now share `_tombstoneFor`/`_bumpAliveOn`/`_tombstoneSlots`/`_reviveSlots`. **Semantic change (accepted):** a concurrent higher-rev remote edit resurrects a deleted element (edit-wins-over-delete); a passive copy stays deleted. Verified the exact partial-eraser round trip is preserved. Covered by `merge_engine_test`, `canvas_controller_test`, `partial_eraser_test`, `split_paste_test`.
