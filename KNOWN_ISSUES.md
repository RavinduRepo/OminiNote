# KNOWN_ISSUES.md

Tracks issues and cross-platform gaps found during codebase audits. Keep this in sync with reality: when an item is fixed, remove it (or move it to "Fixed") instead of leaving it marked done; when new issues are found, add them here rather than only mentioning them in conversation.

## Open issues

### Canvas v1 limitations (deliberate scope cuts — see CANVAS_SPEC.md §17/§19)

- **Eraser is whole-stroke only**; partial/pixel erase later.
- **Selection is single-page**; can't lasso across page boundaries.
- **No camera capture** for images (gallery/file picker only — camera needs a plugin + permissions).
- **Undo/redo history is in-memory** (~100 ops), cleared when the canvas closes.
- **S-Pen button-to-erase works while the pen is touching** (`kPrimaryStylusButton` / inverted stylus); a pure hover-button tool toggle isn't implemented.
- **Orphaned page files**: deleting a page removes it from `section.json` but leaves `pages/<id>.json` on disk (undo may restore it). Harmless; a GC-on-open could clean these later.

### Cloud sync (Drive, v2)

- **Setup prerequisites (not code — Google Cloud config).** Sync needs OAuth clients in the Cloud project: a **Desktop** client (id/secret in `.dart_defines.json`) and an **Android** client registered for package `io.github.ravinduRepo.omininote` + the signing SHA-1 (debug SHA-1 per dev machine for `flutter run`, release SHA-1 `34:48:37:E1:...` for store builds). Each dev machine needs its own Android debug OAuth client registered with its local debug keystore SHA-1. The consent screen is now **In production** with `drive.file` classified as non-sensitive — no user cap, any Gmail can sign in. Branding verification is pending (blocked on `github.io` shared-domain limitation; cosmetic only — app works without it).
- **Open-canvas live merge is now wired** (07/07/26 — see Fixed). Remaining window: two devices editing the *same page* within one poll cycle merge via union on the next round trip — ink is never lost, but LWW-resolved fields (an element's position, page background) can briefly flip-flop before settling.
- **Structural edits are last-writer-wins, not CRDT.** `section.json`/`canvas.json` resolve by `(rev, updatedAt, deviceId)`. Page *ink* is always union-merged and never lost, but if the **same canvas** has pages added/reordered offline on two devices, one device's structural layout wins (the losing side's new page *files* survive on disk/Drive but may be unreferenced). `notebooks.json` avoids this by union-merging per notebook id. Fractional page indexing (plan §1.5) would remove this limitation.
- **No Conflicts Inbox / conflicted copies yet** (plan §3.4). A delete that races an edit resolves by the `(rev,updatedAt,deviceId)` tuple rather than restoring + prompting.
- **GC (30-day tombstone purge — the Recycle bin window) is local-disk only, not Drive-side.** `NotebookService.runGarbageCollection` deletes expired tombstoned notebooks/sections/canvases/pages from local disk (and trims old tombstone entries from live pages), but never calls `DriveService.deleteFile` — Drive's folder API isn't relPath-indexed the way `DriveService`'s file index is, so a safe remote purge needs more plumbing than this pass does. Effect: Drive storage for old deletions isn't reclaimed automatically, and a device that GC'd locally can pull the (still-tombstoned, harmless) file back down on its next full resync. No data-loss risk — the file that comes back is still marked deleted — just a storage/bandwidth cost, not a correctness one.
- **`appDataFolder` not used.** The Drive tree lives in a **visible** `omininote/` folder in My Drive (nice as a user-inspectable backup) under the `drive.file` scope, rather than the hidden `appDataFolder` the plan suggested.

### Multi-account sync (Phase 2 — `SYNC_SHARING_PLAN.md`)

- **Moving a notebook between accounts loses edits made on a device that hasn't synced at move time.** A move fully syncs the notebook DOWN from its source account first (so cloud+local data is preserved) and re-keys it to a new id, but a *third* device that edited the notebook and hasn't uploaded yet will, when it does, write to the now-tombstoned old id → those edits are orphaned. Advise: sync all devices before moving a notebook to another account. (Inherent to any move in an eventually-consistent system.)
- **A moved notebook's old-account content is orphaned on the old Drive.** The re-key tombstones the old notebook id (so other devices drop it), but its page/asset *files* on the source Drive aren't hard-deleted — reclaimed only by the local, 30-day GC, not Drive-side.
- **Moving requires being signed into the notebook's current account on the device doing the move** (to sync it down first). If you aren't, the move is refused rather than risk data loss.
- **Removing an account is per-account but warns on *global* unsynced changes.** The remove-account dialog's "unsynced changes" warning and the gate on the purge checkbox use the whole-journal pending flag, not a per-account count (the dirty journal isn't account-tagged) — so pending edits on account B can block purging account A's local copies. Conservative, not wrong.

### Clipboard / platform

- **Benign `EPIPE (Broken pipe)` warnings on Samsung when copying an image** — logged by `super_native_extensions`'s `DataProvider$PipeDataWriter` (observed live on an SM X510, 07/07/26). One UI's clipboard manager opens the offered image stream repeatedly to build its preview thumbnail and closes the pipe early each time; the plugin catches it and the actual paste target re-requests and receives the full data. Warning noise only — do not chase it as a copy failure.
- **Image clipboard now uses `super_clipboard`** (07/07/26; replaced `pasteboard`) — copy *and* paste of images works on Android, Windows, Linux, macOS, and iOS (the latter two are future targets but already covered). Remaining caveats: on Android both directions depend on the other app using the standard clipboard image contract (all mainstream apps do); Linux still needs a clipboard manager present on very bare setups. Where unsupported, copy/paste falls back to the app-internal clipboard (full fidelity in-app) and OS text. Note `minSdk` is now 23 and first build after `flutter clean` downloads the plugin's precompiled native core (slow once, not broken).

### Notebook sharing (`.omninote` bundles — Phase 3)

- **Samsung "My Files" won't direct-open a `.omninote` file** (verified on an SM X510). Because `.omninote` is a custom, unregistered extension, Samsung's file manager shows "No app can open this" / a Play-Store search *regardless* of the app's `VIEW` intent-filters — it checks against known types before resolving intents. No manifest change fixes it (`application/octet-stream`, `application/zip`, `application/*`, and a path-pattern filter were all tried live). **Workaround: "Share → Omininote" from My Files** (the `SEND` path works everywhere). Direct-tap ("Open with") *does* work from Telegram, Google Files, and most other apps, which send a matchable MIME type. The app deliberately matches `application/zip` + `application/octet-stream` (not `*/*`) to avoid appearing in "Open with" for images/video/etc.
- **`omninote://` share links aren't tappable in messaging/email apps.** Custom URI schemes are only *auto-linkified* by browsers for `http(s)://` — WhatsApp/Telegram/Gmail/etc. render `omninote://import?id=…` as plain, non-tappable text; the recipient has to paste it into a browser address bar to invoke it (which then opens the app and imports fine). The mechanism (host public bundle → download → import) works end to end; the shareability is the limitation. A universally tappable link needs an `https://` **App Link / Universal Link** backed by a hosted domain (a `.well-known/assetlinks.json` + redirect) — deliberately out of scope (no backend). Kept as-is by user choice; "Send a copy" (file / share sheet) is the reliable path.

### Robustness

- **IDs are timestamp-based** (`millis + 3-digit sequence` / `micros + sequence`) — fine in practice, but not UUIDs.
- **Syncfusion licensing**: export builds on `syncfusion_flutter_pdf` under the Community License (user-approved 07/06/26). If the project outgrows eligibility, swap via the `PdfExporter` interface.

### Distribution / installation

- **macOS release build is unsigned.** Distributed `.app` bundles trigger Gatekeeper ("Apple could not verify…") because there is no Developer ID certificate. Users must run `xattr -cr OminiNote.app` or use System Settings → Privacy & Security → Open Anyway on first launch. Resolving requires an Apple Developer account ($99/yr) for Developer ID signing. Mac App Store distribution requires a separate Paid distribution certificate.

- **Android: debug and release builds install as separate apps** if the debug build was ever installed before the package rename (old `com.example.omininote` stays alongside new `io.github.ravinduRepo.omininote`). Users upgrading from a pre-rename debug build must manually uninstall the old one.

### Features — planned

- **Android multi-instance (multiple OminiNote windows).** The goal is running *two or more copies of OminiNote itself* side by side so you can view/edit more than one note at once (Android supports this on large screens / DeX). Basic split-screen *with another app* already works; true multi-instance needs the activity to opt into multiple tasks (`android:resizeableActivity` + a multi-instance/`documentLaunchMode` setup and a way to launch a second instance from Recents), plus verifying the single-instance in-memory state (open controllers, sync listeners) is safe across two processes/tasks. Deferred.

- **Linked copies of sections/canvases.** When duplicating a section, super-section, or canvas, offer a "Linked copy" checkbox: changes in either copy propagate to the other (like OneNote's page sync). Architecture TBD — simplest approach is a `linkId` on the section/canvas level mirroring the existing `TextElement.linkId` pattern.

- **Voice recording + playback sync.** Record audio during a note session; on playback, animate/highlight ink strokes drawn at the same timestamp (like Samsung Notes "Audio Sync"). Requires a recording plugin, timestamp watermarking of strokes, and a playback controller. Large feature — park until core is stable.

- **Note summarisation.** Feed canvas text content to an LLM API and insert a summary text element. Depends on voice recording (for audio transcription path) or can start with text-only canvases.

### Branding

- **`flutter_native_splash` cannot be added to this project right now.** Every version requires either an `xml` range incompatible with `syncfusion_flutter_pdf ^33.2.15`'s `xml ^7.0.1`, or a `meta` version newer than the installed Flutter SDK's bundled `meta 1.17.0` — confirmed via `flutter pub add --dry-run`, a genuine current ecosystem conflict, not a config mistake. Re-check if either package's constraints loosen in a future update.
- **No native splash screen on Windows/macOS/Linux.** Android's is hand-implemented (see CLAUDE.md's Branding section); desktop splash screens would require editing native runner code (C++/Swift), not just resource files, so it was left out of scope. Flag if you want it — it's a bigger, separate effort than the Android side was.
- **iOS and web app icons are intentionally skipped** in the `flutter_launcher_icons` config (`ios: false`, no `web` block) — consistent with iOS/web not being project priorities.

- ~~Eraser-deleted ink came back after sync; edited/moved elements never converged ("latest should win"); new pages only arrived via manual "Repair sync"~~ (07/07/26) — three distinct bugs: **(1) The eraser tool never wrote tombstones on the actual erase.** `_commitErase`'s first-apply replay guard skipped the tombstone-writing block (the live gesture had only removed strokes from the array); tombstones were only written on *redo*. So a plain erase synced as "stroke missing, no tombstone" → the other device's union merge restored it → "deleted line comes back appended to the latest changes". Guard removed — apply is idempotent anyway (`removeWhere` on already-removed ids is a no-op). Lasso-delete had been fixed earlier; the eraser tool was the remaining (and primary) hole. **(2) Element edits never bumped `rev`/`updatedAt`.** Moving/resizing/recoloring/retyping an element left its envelope at rev 1/creation-time, so both devices' copies tied in the `(rev, updatedAt, deviceId)` comparison and each kept its own version forever — no convergence, endless mutual re-pushes. Now every mutating op commit stamps the affected elements (`_stamp` → `bumpRev`): transform, text style, recolor, text edit, cross-page move. Known edge: undoing an already-synced edit restores the *older* rev, which loses LWW to the synced copy — the undo may be reverted by the next pull (rare; documented tradeoff). **(3) Files created by another device only arrived via full resync, and even that raced.** Unknown fileId in the changes feed → the code scheduled a *full-tree* resync (with the token advanced past the change regardless — if that resync was skipped/raced, the file was lost until a manual Repair), and `fullResync` took its new changes baseline *after* the listing, silently skipping anything pushed during a slow resync. Now: `DriveService.resolveFileId` places an unknown file by walking its parent chain (couple of `files.get`s, folder-cache-accelerated, recording the mapping with **no** head so echo suppression can't skip the first download) and pulls it inline; `fullResync` takes the baseline token *before* listing; and `_pullFile` skips re-downloading files whose `headRevisionId` matches the last reconciled one (with `listAllFiles` careful to *preserve* previously-reconciled heads rather than stamping current ones — otherwise the resync would skip everything), so Repair now scales with what changed, not with total data size. Covered by new tests: eraser-first-apply tombstones + transform rev-stamping (`test/canvas_controller_test.dart`), higher-rev-copy-wins (`test/merge_engine_test.dart`).
- ~~Edits inside an open canvas resurrected after sync (deletes came back, remote strokes vanished); moving elements between pages duplicated them; a fresh canvas took ages to reach the other device~~ (07/07/26) — three separate bugs: **(1) Open-canvas stale-state clobber (the resurrection loop):** the pull pipeline merged remote changes onto *disk* only; an open canvas's in-memory controller never saw them, so its next 500 ms autosave overwrote the merged file *without* the other device's tombstones and re-pushed it — each device kept re-uploading exactly what the other kept deleting, forever. Fixed by wiring live merge: `SyncService` now routes pulled page/canvas files of an open canvas to its `CanvasController` (`registerCanvasListener` → `applyRemotePage`/`applyRemoteStructure`), which set-union-merges into the live in-memory page (same instance, so painter/undo stay valid), drops remotely-tombstoned pages, and only marks itself dirty when the union actually contains something Drive lacks (pushing merged==remote would just echo revisions back and forth). **(2) Cross-page move left no tombstone:** `_reparentSelection` cleared elements off the source page with a bare rebuild, so a stale remote copy of the source page resurrected them → duplicates (also live on the target). Now the move op tombstones the moved ids on the source (and clears any old tombstones for them on the target); undo reverses both. **(3) Push was slow:** every upload of a not-yet-indexed file resolved its whole Drive folder chain with one sequential `files.list` per segment — *twice* (find + ensure-parent). A new canvas pushes 4+ files ≈ 50 sequential API round trips. Now folder ids are cached (`_folderIds`, also warmed from `listAllFiles` during resync), the path is walked once, and a stale-index 404 on update falls back to create. Poll interval also halved (30 s → 15 s). Covered by new tests in `test/canvas_controller_test.dart` (deleteSelection tombstones + undo; applyRemotePage union/erase/page-tombstone).
- ~~Deletions "didn't stick" — a notebook/section/canvas/page/text-box/image deleted on one device would come back after sync unless every other device also deleted it~~ (07/07/26) — root cause: the envelope fields (`rev`/`updatedAt`/`deviceId`/`deletedAt`) existed on every model, but the service layer still **hard-deleted** (removed the map key, recursively deleted the directory, spliced the array) instead of ever writing the tombstone. A hard-removed entry has no envelope left to lose a comparison with, so any device that still held a live copy won the union merge and brought it back — sometimes for weeks, until the last holdout finally deleted it too. Fixed by making every delete a **soft-delete**: `deleteNotebook`/`deleteSection`/`deleteCanvas` now set `deletedAt`+bump `rev` and keep the entry/file on disk; `getNotebooks`/`getNotebook`/`getSection`/`getCanvas` filter `deletedAt != null` so the UI still sees it as gone. Text/image object deletes got the same fix via a new `CanvasPage.deletedObjects` tombstone list (mirroring the `erased` list strokes already had) — `deleteSelection`/`removeElement` now tombstone instead of a bare array splice (this also fixed a **second**, pre-existing bug: lasso-delete of a *stroke* never added it to `erased[]` either — only the eraser tool did — so lasso-deleted ink could resurrect too). Page deletion now explicitly persists the page's own tombstone (it leaves the in-memory `pages` map before the normal autosave would ever catch it) and `NotebookService.loadPages` prunes any row still pointing at a tombstoned page, self-healing a canvas.json structural race. A 90-day GC sweep (`NotebookService.runGarbageCollection`, local-disk only) now physically purges expired tombstones. Covered by `test/merge_engine_test.dart` (object-tombstone-vs-live-remote-copy, soft-deleted-notebook-beats-live-remote-copy) and a model round-trip in `test/models_test.dart`.
