# KNOWN_ISSUES.md

Tracks issues and cross-platform gaps found during codebase audits. Keep this in sync with reality: when an item is fixed, remove it (or move it to "Fixed") instead of leaving it marked done; when new issues are found, add them here rather than only mentioning them in conversation.

## Open issues

### Canvas v1 limitations (deliberate scope cuts ŌĆö see CANVAS_SPEC.md ┬¦17/┬¦19)

- **Selection is single-page**; can't lasso across page boundaries.
- **Undo/redo history is in-memory** (~100 ops), cleared when the canvas closes.
- **S-Pen button-to-erase works while the pen is touching** (`kPrimaryStylusButton` / inverted stylus); a pure hover-button tool toggle isn't implemented.

### Cloud sync (Drive, v2)


- **Setup prerequisites (not code ŌĆö Google Cloud config).** Sync needs OAuth clients in the Cloud project: a **Desktop** client (id/secret in `.dart_defines.json`) and an **Android** client registered for package `io.github.ravinduRepo.omininote` + the signing SHA-1 (debug SHA-1 per dev machine for `flutter run`, release SHA-1 `34:48:37:E1:...` for store builds). Each dev machine needs its own Android debug OAuth client registered with its local debug keystore SHA-1. The consent screen is now **In production** with `drive.file` classified as non-sensitive ŌĆö no user cap, any Gmail can sign in. Branding verification is pending (blocked on `github.io` shared-domain limitation; cosmetic only ŌĆö app works without it).
- **Open-canvas live merge is now wired** (07/07/26 ŌĆö see Fixed). Remaining window: two devices editing the *same page* within one poll cycle merge via union on the next round trip ŌĆö ink is never lost, but LWW-resolved fields (an element's position, page background) can briefly flip-flop before settling.
- **Structural edits are last-writer-wins, not CRDT.** `section.json`/`canvas.json` resolve by `(rev, updatedAt, deviceId)`. Page *ink* is always union-merged and never lost, but if the **same canvas** has pages added/reordered offline on two devices, one device's structural layout wins (the losing side's new page *files* survive on disk/Drive but may be unreferenced). `notebooks.json` avoids this by union-merging per notebook id. Fractional page indexing (plan ┬¦1.5) would remove this limitation.
- **No Conflicts Inbox / conflicted copies yet** (plan ┬¦3.4). A delete that races an edit resolves by the `(rev,updatedAt,deviceId)` tuple rather than restoring + prompting.
- **A purge permanently discards unsynced offline edits to the purged item.** Inherent to "completely wipe": a device that edited a notebook offline and reconnects after the notebook was purged has its uploads for that subtree dropped by the push filter ŌĆö the edits are gone (the alternative would repopulate Drive with orphaned content). The 30-day bin window before auto-purge is the mitigation.
- **`appDataFolder` not used.** The Drive tree lives in a **visible** `omininote/` folder in My Drive (nice as a user-inspectable backup) under the `drive.file` scope, rather than the hidden `appDataFolder` the plan suggested.

### Multi-account sync (Phase 2 ŌĆö `SYNC_SHARING_PLAN.md`)

- **Moving a notebook between accounts loses edits made on a device that hasn't synced at move time.** A move fully syncs the notebook DOWN from its source account first (so cloud+local data is preserved) and re-keys it to a new id, but a *third* device that edited the notebook and hasn't uploaded yet will, when it does, write to the now-tombstoned old id ŌåÆ those edits are orphaned. Advise: sync all devices before moving a notebook to another account. (Inherent to any move in an eventually-consistent system.)
- **A moved notebook's old-account content lingers on the old Drive for up to 30 days.** The re-key tombstones the old notebook id (so other devices drop it); when that tombstone expires, the GC's purge pass now deletes the old Drive folder too ŌĆö but only from a device signed into the *old* account at the time. If no device ever signs into that account again, its orphaned folder stays.
- **Moving requires being signed into the notebook's current account on the device doing the move** (to sync it down first). If you aren't, the move is refused rather than risk data loss.
- **Removing an account is per-account but warns on *global* unsynced changes.** The remove-account dialog's "unsynced changes" warning and the gate on the purge checkbox use the whole-journal pending flag, not a per-account count (the dirty journal isn't account-tagged) ŌĆö so pending edits on account B can block purging account A's local copies. Conservative, not wrong.

### Clipboard / platform

- **Benign `EPIPE (Broken pipe)` warnings on Samsung when copying an image** ŌĆö logged by `super_native_extensions`'s `DataProvider$PipeDataWriter` (observed live on an SM X510, 07/07/26). One UI's clipboard manager opens the offered image stream repeatedly to build its preview thumbnail and closes the pipe early each time; the plugin catches it and the actual paste target re-requests and receives the full data. Warning noise only ŌĆö do not chase it as a copy failure.
- **Image clipboard now uses `super_clipboard`** (07/07/26; replaced `pasteboard`) ŌĆö copy *and* paste of images works on Android, Windows, Linux, macOS, and iOS (the latter two are future targets but already covered). Remaining caveats: on Android both directions depend on the other app using the standard clipboard image contract (all mainstream apps do); Linux still needs a clipboard manager present on very bare setups. Where unsupported, copy/paste falls back to the app-internal clipboard (full fidelity in-app) and OS text. Note `minSdk` is now 23 and first build after `flutter clean` downloads the plugin's precompiled native core (slow once, not broken).

### Notebook sharing (`.omninote` bundles ŌĆö Phase 3)

- **Samsung "My Files" won't direct-open a `.omninote` file** (verified on an SM X510). Because `.omninote` is a custom, unregistered extension, Samsung's file manager shows "No app can open this" / a Play-Store search *regardless* of the app's `VIEW` intent-filters ŌĆö it checks against known types before resolving intents. No manifest change fixes it (`application/octet-stream`, `application/zip`, `application/*`, and a path-pattern filter were all tried live). **Workaround: "Share ŌåÆ Omininote" from My Files** (the `SEND` path works everywhere). Direct-tap ("Open with") *does* work from Telegram, Google Files, and most other apps, which send a matchable MIME type. The app deliberately matches `application/zip` + `application/octet-stream` (not `*/*`) to avoid appearing in "Open with" for images/video/etc.
- **`omninote://` share links aren't tappable in messaging/email apps.** Custom URI schemes are only *auto-linkified* by browsers for `http(s)://` ŌĆö WhatsApp/Telegram/Gmail/etc. render `omninote://import?id=ŌĆ”` as plain, non-tappable text; the recipient has to paste it into a browser address bar to invoke it (which then opens the app and imports fine). The mechanism (host public bundle ŌåÆ download ŌåÆ import) works end to end; the shareability is the limitation. A universally tappable link needs an `https://` **App Link / Universal Link** backed by a hosted domain (a `.well-known/assetlinks.json` + redirect) ŌĆö deliberately out of scope (no backend). Kept as-is by user choice; "Send a copy" (file / share sheet) is the reliable path.

### Robustness

- **IDs are timestamp-based** (`millis + 3-digit sequence` / `micros + sequence`) ŌĆö fine in practice, but not UUIDs.
- **Syncfusion licensing**: export builds on `syncfusion_flutter_pdf` under the Community License (user-approved 07/06/26). If the project outgrows eligibility, swap via the `PdfExporter` interface.

### Distribution / installation

- **macOS release build is unsigned.** Distributed `.app` bundles trigger Gatekeeper ("Apple could not verifyŌĆ”") because there is no Developer ID certificate. Users must run `xattr -cr OminiNote.app` or use System Settings ŌåÆ Privacy & Security ŌåÆ Open Anyway on first launch. Resolving requires an Apple Developer account ($99/yr) for Developer ID signing. Mac App Store distribution requires a separate Paid distribution certificate.

- **Android: debug and release builds install as separate apps** if the debug build was ever installed before the package rename (old `com.example.omininote` stays alongside new `io.github.ravinduRepo.omininote`). Users upgrading from a pre-rename debug build must manually uninstall the old one.

### Features ŌĆö planned

- **Android multi-instance (multiple OminiNote windows).** The goal is running *two or more copies of OminiNote itself* side by side so you can view/edit more than one note at once (Android supports this on large screens / DeX). Basic split-screen *with another app* already works; true multi-instance needs the activity to opt into multiple tasks (`android:resizeableActivity` + a multi-instance/`documentLaunchMode` setup and a way to launch a second instance from Recents), plus verifying the single-instance in-memory state (open controllers, sync listeners) is safe across two processes/tasks. Deferred.

- **Linked copies of sections/canvases.** When duplicating a section, super-section, or canvas, offer a "Linked copy" checkbox: changes in either copy propagate to the other (like OneNote's page sync). Architecture TBD ŌĆö simplest approach is a `linkId` on the section/canvas level mirroring the existing `TextElement.linkId` pattern.

- **Voice recording + playback sync.** Record audio during a note session; on playback, animate/highlight ink strokes drawn at the same timestamp (like Samsung Notes "Audio Sync"). Requires a recording plugin, timestamp watermarking of strokes, and a playback controller. Large feature ŌĆö park until core is stable.

- **Note summarisation.** Feed canvas text content to an LLM API and insert a summary text element. Depends on voice recording (for audio transcription path) or can start with text-only canvases.

### Branding

- **`flutter_native_splash` cannot be added to this project right now.** Every version requires either an `xml` range incompatible with `syncfusion_flutter_pdf ^33.2.15`'s `xml ^7.0.1`, or a `meta` version newer than the installed Flutter SDK's bundled `meta 1.17.0` ŌĆö confirmed via `flutter pub add --dry-run`, a genuine current ecosystem conflict, not a config mistake. Re-check if either package's constraints loosen in a future update.
- **No native splash screen on Windows/macOS/Linux.** Android's is hand-implemented (see CLAUDE.md's Branding section); desktop splash screens would require editing native runner code (C++/Swift), not just resource files, so it was left out of scope. Flag if you want it ŌĆö it's a bigger, separate effort than the Android side was.
- **iOS and web app icons are intentionally skipped** in the `flutter_launcher_icons` config (`ios: false`, no `web` block) ŌĆö consistent with iOS/web not being project priorities.

- ~~Sync repeatedly threw `FormatException: Unexpected extension byte` on full resync after using "Share a link"~~ (07/15/26) ŌĆö the resync loop pulled **every** file under the Drive `omininote/` folder, including the public `omininote/shared/*.omninote` bundles that "Share a link" uploads. - ~~Importing a large `.omninote` bundle could crash on Android (OOM)~~ (07/15/26) ŌĆö **streaming import** (#4b, symmetric to the 07/14 streaming export). - ~~Move an item to another page and back deleted it; and undo/fresh-id revival broke the undo chain~~ (07/12/26) ŌĆö **rev-based (LWW) deletion for strokes & objects**, unifying them with how the Recycle bin already deletes notebooks/sections/canvases.
