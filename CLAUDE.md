# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Omininote (`omininote`) is a Flutter stylus note-taking app. Structure: **Notebook → Section → Canvas**, where a Section opens into a freely pan/zoomable **Canvas** of many **pages** (blank or PDF-backed) that the user inks, types, and pastes images onto — Samsung-Notes-style, with vector PDF export. Target platforms are Android, Windows, macOS, and Linux (iOS is not a priority; web is not configured). The full feature spec is `CANVAS_SPEC.md`; the user's raw intent notes are `PAGE_BEHAVIOR_REQ.md`; open bugs live in `KNOWN_ISSUES.md` — keep all of these in sync with code changes.

## Commands

```bash
flutter pub get                 # install dependencies (run after editing pubspec.yaml)
flutter analyze                 # static analysis / lints
flutter test                    # run all tests
flutter test test/models_test.dart    # run a single test file
flutter devices                 # list run targets (physical devices show here even if adb isn't on PATH)
flutter run -d <device-id>      # run on a device from `flutter devices`
flutter run -d windows          # run desktop build directly
flutter build apk / windows / linux / macos
```

Sync builds need the gitignored `.dart_defines.json` (OAuth client ids; template in `.dart_defines.example.json`, real values in `SECRETS/`): `flutter run -d <id> --dart-define-from-file=.dart_defines.json`. The `Makefile` wraps this per platform (`make android|linux|windows|macos`); on Windows invoke it as `mingw32-make` from Git Bash.

No custom lint config beyond default `package:flutter_lints/flutter.yaml`.

**OneNote importer** (`tools/onenote_importer/`, own README): recovers `.onepkg`/`.one` notes into the app's on-disk store — a Rust extractor built on Joplin's OneNote parser (source npm-fetched into gitignored `.cache/`) dumps `extract.json` + lossless assets, `preview.html` visualizes it, and `convert.dart` (pure Dart, run via `dart run`) emits/installs the store. Section groups → notebook `FolderNode`s, sub-pages → canvas-tree `FolderNode`s, and each infinite OneNote page is tiled into rows/pages with cuts only where no element crosses. Covered by `test/onenote_import_test.dart` (tiling invariants + model round-trip of converted output).

## Architecture

Single-package Flutter app, no state-management library (plain `ChangeNotifier` + `setState`), no backend — persistence is local JSON files. Key dependencies: `pdfrx` (PDF page **rendering only** — its viewer/gestures are not used), `perfect_freehand` (ink outlines), `syncfusion_flutter_pdf` (vector PDF export; Community License), `file_picker`, `path_provider`, `crypto` (content-addressed assets), `super_clipboard` (OS image/HTML clipboard), `html` (parsing pasted rich-text HTML into styled runs).

**Navigation (mobile — bottom-nav shell):** `MobileShellScreen` (`lib/screens/mobile_shell_screen.dart`) is the mobile root (picked by `_RootRouter`). It's a **bottom navigation bar** over four destinations — **Notebooks / Search / Bin / Settings** — where each tab owns its **own nested `Navigator`** (a `GlobalKey<NavigatorState>` per tab, kept alive via a `PageView` + `AutomaticKeepAliveClientMixin` wrapper). **Tabs slide horizontally**: tapping a tab `animateToPage`s toward it, and a horizontal **swipe** moves between tabs — but only at a tab's root (a per-tab `NavigatorObserver` recomputes `_swipeEnabled` post-frame — never synchronously, or the initial-route push at startup would `setState` during build — and locks `PageView` physics to `NeverScrollableScrollPhysics` once you've drilled in, so drags belong to the content). The notebook drill-down (`HomeScreen` notebooks → `NotebookScreen` sections → `SectionScreen` canvases) pushes **inside the Notebooks tab's** nested navigator via **`slideRoute`** (horizontal slide-in from the right + parallax-dim of the page below — `app_theme.dart`, distinct from the `fadeThroughRoute` used elsewhere), so the bottom bar **stays visible the whole way down**. The **Bin tab reloads on entry** (`BinScreen.refreshSignal` — a `ValueNotifier` the shell bumps whenever the Bin tab is opened; the kept-alive tab's `initState` load otherwise never re-ran, so items deleted elsewhere wouldn't appear). **Perf (07/13/26):** that entry-reload used to rescan the whole store (`listDeletedItems` walks every notebook/section/canvas + a head-read of every page file) *every* time — the main cause of swipe-into-Bin jank. Now (a) the shell **debounces the `refreshSignal` bump ~350ms** (a `Timer` in `_onEnterTab`) so the scan can't compete with the slide animation, and (b) `BinScreen` **caches its list and only rescans when the store changed** — it captures `SyncService.dataVersion` before each scan and `_maybeReload` skips the scan unless `dataVersion` advanced. For this to stay correct, **local** delete/restore/purge now bump `dataVersion` too (previously only remote pulls did): `notifyDataChanged()` is called from `deleteNotebook`/`deleteSection`/`deleteCanvas`/`deleteSectionFolder`/`deleteCanvasFolder`/`restoreBinItem`/`purgeBinItem`, and from `savePage` **only when the page carries a tombstone** (`deletedAt != null` — catches `CanvasController.deletePage` without firing on ordinary drawing autosaves or the GC's `_writeAtomic`-direct purge). The first load (and the Settings→Deleted-items path, which has no `refreshSignal`) runs via a post-frame callback so it's off the push-animation frame. Opening a canvas pushes `CanvasScreen` on the **root** navigator (`Navigator.of(context, rootNavigator: true)`), which covers the shell → **the editor is full-bleed with no bottom bar** (the deliberate exception; everything else keeps the bar). The bar is a custom `_MobileTabBar` (neutral surface + hairline top border, icon-over-label going amber when active — no Material pill). `PopScope` routes the system back button: pop the active tab's nested navigator first, else jump to the Notebooks tab. **Search is now a tab** (`NoteSearchView`, the embeddable form of the search screen — rebuilds its index on `SyncService.dataVersion`, **debounced 400ms** so a burst of changes coalesces into one full-store reindex; **07/13/26:** local create/rename (`createNotebook/Section/Canvas`, `createSectionFolder/CanvasFolder`, `renameNotebook/Section/Canvas`) now bump `dataVersion` too — previously only remote pulls + deletes did — so a freshly-**added** item is searchable without an app restart; the field focuses only on tab entry (`focusSignal`) and is **unfocused when you leave the Search tab (`onPageChanged` to a non-Search tab) or open a result (`_open`)**, so the keyboard never re-pops when returning to a list from a canvas; an **empty query renders a hint, not the whole index** — `_results` is `[]` until you type; the per-keystroke `filter` is **debounced 140ms for large indexes** (`≥400` results) and instant below that, with a flush-on-submit so Enter opens a current result); `SearchService.buildIndex` reads its section/canvas files **bounded-concurrently** (`_mapBounded`, 8 in flight) instead of sequentially — the small structure-only decode stays on-main (offloading wouldn't pay: the models must be built here to navigate to results). **Type filters + path matching (07/19/26):** the search screen shows a chip row (Notebooks / Sections / Canvases / Bookmarks, all on = no restriction) whose selection maps to a `Set<SearchKind>` passed to `SearchService.filter(..., kinds:)`; each `SearchResult` precomputes `titleLower` + `pathKeyLower` (breadcrumb + title) at index time, and `filter` matches the title first, then — for 3+ char queries — the full path key, so a run-together path like "noteseccanv" surfaces Notebook › Section › Canvas (path-only hits always rank below title hits via `_kPathMatchPenalty`). Covered by `test/search_service_test.dart`. picking a result calls the shell's `_revealSearchResult`, which switches to the Notebooks tab, rebuilds its drill-in stack (glow trail preserved via `glowId`), and opens any canvas on the root navigator — mobile now uses the same `onReveal` hook the desktop shell always did (no more root-navigator `popUntil` rebuild). **Bin is a tab** too (the Settings → Deleted items row still exists as a secondary path). Search/Settings are therefore **gone from the list screens' app bars** (`HomeScreen`/`NotebookScreen`/`SectionScreen` keep only the `+`). All list screens: **no FABs**; adding goes through the app-bar `+`. Each list supports **pull-to-refresh** (`RefreshIndicator` → `SyncService().syncNow()`; `RefreshableEmpty` keeps the gesture on empty states). The canvas itself has no pull-to-refresh. **Note:** `NotebookScreen`/`SectionScreen` still `searchRouteObserver.subscribe` for glow-on-pop-back, but that observer is on the *root* navigator while they now live in a nested one, so the re-glow-on-return trail no longer fires (the initial `glowId` glow still does) — an accepted minor regression of the nested-navigator shell. **Mobile kebab menus are bottom sheets:** `showActionSheet` (`lib/widgets/action_sheet.dart`) renders a **plain edge-to-edge** sheet (grab handle, icon+label rows, destructive in the error color) using the theme's own background + shape — the SAME look as the "+" (Add) sheet, via `scrollableSheetBody`. It deliberately does **not** draw its own floating bordered card any more: that card, inside the theme's already-bordered sheet, produced a double vertical line down each side ("weird two lines" — fixed 07/19/26); keep it borderless so every sheet reads as one consistent outline. `ItemTreeView`'s leaf/folder kebabs use it when `!dense` (mobile) and keep `PopupMenuButton` when `dense` (desktop), home's notebook `_RowMenu` always uses it (mobile-only), and the canvas overflow menu uses it when `!widget.embedded`. **All modal bottom sheets are scroll-safe on small screens** (`action_sheet.dart` also exports `scrollableSheetBody` = SafeArea + a `kSheetMaxHeightFactor`=0.9 height cap + `SingleChildScrollView`, and `cappedSheetBody` = the same cap *without* the outer scroll view, for sheets that own an inner `Flexible`+`ListView`); every `showModalBottomSheet` call passes `isScrollControlled: true` and wraps its body in one of these (fixed-content sheets — Add / insert-position / PDF-mode / Page settings / color picker — use `scrollableSheetBody`; the bookmarks + attachments list sheets use `cappedSheetBody`). Add new sheets the same way or tall content overflows on short screens (this was a real bug once Page settings grew the ink-visibility section). **Bin + search rows** carry the same colored identity rail as the tree (bin: `AppPalette.resolveColor` of the item's most-specific id; search: canvas→section→notebook).

### Responsive shell (`main.dart`'s `_RootRouter`, `lib/screens/desktop_shell_screen.dart`)

`MaterialApp.home` is `_RootRouter`, which picks between the mobile push-navigation flow above and `DesktopShellScreen` — a OneNote-desktop-style view with a **60px left icon nav rail** (`_buildNavRail`: logo, then Notebooks / Search / Bin destinations, with Settings + an account avatar pinned to the bottom — the v2 redesign's rail; `palette.canvas` background) followed by **three panes**. The rail is wired to existing behavior (per the user's scope choice, not a pane-switcher rework): **Notebooks** toggles the left panes collapsed/expanded (mirrors the header book icon), **Search** opens the overlay, **Bin/Settings** push their screens — so Search + Settings were removed from the sidebar header, which keeps only the collapse toggle + `+` + sync. The build's pane-width math subtracts the 60px rail from `constraints.maxWidth`. The three panes: a resizable left sidebar showing the notebook → super-section → section tree (see "Item tree" below), a **resizable canvas-list column** (`_buildCanvasColumn`, OneNote's page list; default 230px, drag-resizable 170–380px via a second `_resizeDivider`; `palette.surface2` background, dense `ItemTreeView<Canvas>`, slides closed via `AnimatedContainer` when nothing is selected — the divider only renders while open), and `CanvasScreen` embedded directly in the main pane (no pushed route, no breadcrumb — the list stays visible beside it; panes are keyed by canvas id so state resets cleanly). Notebooks load into state as mutable `Notebook` objects (the in-memory source of truth: tree drags/collapses mutate them in place then `saveNotebook`); operations that touch section *files* call `_reloadNotebook(id)` to resync. The header's book icon (`_buildSidebarToggle`) collapses **both** left panes (`_sidebarCollapsed` shrinks the sidebar to a narrow rail and slides the canvas column to width 0) showing just a colored initial per notebook — tapping one expands back and jumps straight to it; the resize-drag divider only renders while expanded. **Collapse state is preserved (07/20/26):** `_sidebarCollapsed` is the user's *explicit* choice and is kept independent of two things that used to clobber it — canvas **full screen** now tracks its own `_fullScreen` flag (`onFullScreenChanged` sets it, reset in `_openMainMode` since leaving the canvas view disposes the CanvasWorkspace so its exit callback can't fire), and the **Search/Bin/Settings full-pane modes** just switch `_mainMode` without collapsing. The layout hides the panes whenever `_mainMode != canvas || _fullScreen || _sidebarCollapsed` (`panesVisible`/`sidebarShown` in `build`), so returning from a mode or exiting full screen restores exactly the collapse the user left; only pressing the Notebooks rail *while already in canvas mode* toggles `_sidebarCollapsed` (explicit navigation like link/search reveal + create-notebook still expand deliberately). **Pin canvas list (`_canvasListPinned`):** a push-pin toggle in the canvas-list header keeps the canvas-list column visible even when the notebook sidebar is collapsed — `canvasColumnOpen = panesVisible && section != null && (!_sidebarCollapsed || _canvasListPinned)`. **Animation gotcha (fixed):** the sidebar content is laid out at its *target* width inside `ClipRect`+`OverflowBox` so the width animation clips/reveals rather than reflowing the children mid-transition (an unclipped `AnimatedContainer` reflowed the header `Row` at intermediate widths → visible overflow artifacts on expand). A 20px top inset keeps the toggle off the very top-left window corner. The choice is `LayoutMode.auto` (window width ≥ 840 logical px via `LayoutBuilder`, live on resize), or a manual `mobile`/`desktop` override — both persisted via `SettingsService.layoutMode` and editable from `SettingsScreen`'s "Layout" section. `CanvasScreen` takes an optional `onSectionRenamed` callback so the sidebar can invalidate its cached section list when a rename happens from *inside* the embedded canvas (the sidebar's `NotebookService.getSections()` call returns fresh `Section` objects on each fetch, decoupled from the mutable instance `CanvasScreen` edits in place — without the callback the sidebar would show the stale name). **Gotchas:** never write `setState(() => someMap.remove(key))` as a single-expression arrow body when the map's value type is a `Future` — the arrow's implicit return value becomes that `Future`, and `State.setState` asserts against callbacks that return one (`"setState() callback argument returned a Future"`). Use a block body (`setState(() { someMap.remove(key); })`) instead. Also: the sidebar's `AnimatedContainer` width is wrapped in a `LayoutBuilder` that clamps it to `constraints.maxWidth` — don't remove that clamp, since some Android devices briefly report a near-zero-width frame at startup, and an unclamped fixed-width sidebar overflows its `Row` for that frame (found on a real tablet, not just theoretical). **Perf (07/14/26):** (a) `_selectSection` caches each section's canvas map in `_canvasCache` (keyed by section id) so re-selecting a visited section is instant (no disk re-read); the cache is **cleared on every `dataVersion` bump** (`_onSyncData`) and **refreshed on local canvas edits** (`_reloadSelectedSection` rewrites its entry; `_relocateCanvas` drops the destination section's entry) so it can't serve a stale list across a sync or a mutation. (b) A notebook's section tree is built **only once it's been opened** — `_everExpanded` (a superset of `_expanded`, added-to at every expand site: the row toggle, `_revealSearchResult`, `_createNotebook`, `_importNotebook`) gates the expand `AnimatedCrossFade`'s `firstChild` via `expanded || _everExpanded.contains(id)`. `AnimatedCrossFade` builds **both** children regardless of collapse state, so without the gate every notebook's whole section subtree was (re)built on every sidebar `setState` (including each canvas click, which calls the shell's `setState`); the gate makes sidebar rebuild cost scale with *opened* notebooks, not total sections. The `expanded ||` half guarantees the real tree is present whenever the animation actually needs it, so animations are unchanged. (`_loadAll` still eagerly loads every notebook's section map — now bounded-concurrently — so expanding stays instant; making that lazy too was deferred as higher-risk.)

### Theming (`lib/theme/app_theme.dart`)

All colors/shapes flow through the `AppPalette` `ThemeExtension` (read via `Theme.of(context).extension<AppPalette>()!`) built by a **`ThemeVariant`** (`app_theme.dart`) — a named palette + its scaffold/surface/onSurface/onAccent, with `.build()` → `ThemeData` via the shared `AppTheme._build`. **Selectable palettes (07/18/26):** `AppTheme.lightVariants` = Slate (default, cool grey+amber) / Paper (warm ivory) / Mist (cool blue) / Sage (soft green); `AppTheme.darkVariants` = Charcoal (default, deep grey+amber, the v2 mockup) / Midnight (navy+blue) / Espresso (warm dark+amber) / Carbon (near-black+teal). All tuned easy-on-the-eye for long reading; **never pure black**. Rounder cards (`kRadius`=10 — was 6; the "v2 redesign" softened the old crisp Graphite shapes). `SettingsService` persists **`lightThemeId` + `darkThemeId` independently** (`settings.json`), and `main.dart`'s MaterialApp sets `theme` = `AppTheme.lightVariant(lightThemeId).build()`, `darkTheme` = `AppTheme.darkVariant(darkThemeId).build()`, `themeMode` = the mode — so **System shows the chosen light in light and the chosen dark in dark**, each remembered on its own (a manual Light/Dark forces one). Settings → Appearance shows two rows of live palette-preview chips (`_VariantChip`, a mini note-card swatch in the variant's own colors) under the System/Light/Dark control. `AppTheme.light()/dark()` remain as the defaults (first variant). Also a `ValueNotifier<ThemeMode>` (live) **and** `ValueNotifier<PageBackground> defaultPageBackground` (app-wide default page color + pattern); both are edited from `SettingsScreen`, and `createSection` seeds a new section's default background from the latter. Dates format MM/DD/YY and counts get thousands separators via `lib/utils/formatting.dart`.

### Item tree (`lib/widgets/item_tree_view.dart`, `color_swatch_picker.dart`)

`ItemTreeView<T>` renders a container's recursive contents (leaf items + nested super-section folders) as a drag-reorderable, collapsible tree; it's generic and reused at two levels — sections inside a notebook (desktop sidebar `dense: true`, mobile `NotebookScreen` `dense: false`) and canvases inside a section (desktop canvas-list column `dense: true` + mobile `SectionScreen`). It flattens the tree (respecting each group's `collapsed`), renders each row as a `LongPressDraggable`, and places `DragTarget` insertion-gaps between rows (reorder within a parent) plus makes each group header a `DragTarget` (drop *into* the group, at its end); a drop whose `containerId` differs routes to `onCrossDrop`. It mutates the passed-in `nodes` **in place** then calls `onTreeChanged` to persist; all other actions (rename/color/delete/add/open/relocate) are delegated to callbacks so the host owns dialogs + service calls. `_wouldCycle` blocks dropping a group into its own subtree. The view returns a shrink-wrapping `Column` (not a scrollable) so it composes inside each expandable notebook within the host's single scroll view. Colors are set via `showColorSwatchPicker` (curated `AppPalette.swatchColors`; returns a `ColorChoice` whose null `color` means "use the default identity color"). **Visual hierarchy** (deliberate, both themes, OneNote-style — *rows stay neutral, color lives only in small elements*; full-row color washes were tried and rejected as "too much color"): folder rows = chevron + filled folder icon in the folder's color + w600 text on a transparent row; leaf rows are lighter — a short colored identity pill (3px, indents with the tree) + dim icon + w400 text; nesting depth is drawn as hairline vertical indent guides (`_indentGuides`) — don't collapse these back into uniform rows, telling the levels apart was a user complaint. Notebook-level reordering is a separate `ReorderableListView` (`buildDefaultDragHandles: false`) with the whole notebook row/card wrapped in `ReorderableDelayedDragStartListener` — long-press to reorder, **no visible drag handle** (the old `drag_indicator` icons looked messy; notebook headers aren't tree rows, so this doesn't fight the tree's `LongPressDraggable`). Desktop sidebar notebook rows carry a 20px solid initial chip (the only colored element; row bg is `surface2` while expanded, else transparent); the expanded section tree below is indented under a 2px rail in the notebook's color. Mobile home cards use a 38px solid initial chip on a neutral card.

### Data model (`lib/models/`)

All geometry is stored in **PDF points** (1pt = 1/72"; default page A4 595×842) so export maps 1:1.

- `Notebook`: id, name, createdAt, optional `color` (ARGB int; null → deterministic `AppPalette.identityColor`), and a **recursive contents tree** `nodes: List<NotebookNode>`. `NotebookNode` (sealed, in `notebook.dart`) = `SectionNode {sectionId}` (leaf) | `GroupNode {id, name, color?, collapsed, children}` — a **super-section** (OneNote "section group") that holds sections *and* nested groups, arbitrarily deep. `allSectionIds` flattens depth-first. Legacy notebooks with a flat `sectionIds` list auto-upgrade to a flat tree of `SectionNode`s on load; `toJson` still writes a derived flat `sectionIds` for any back-compat reader.
- `Section` (`section.dart`): the document; owns `rows: [PageRow]` (each row = horizontal band of `pageIds`), `defaultPageSize/Background`, `attachments`, optional `color` (like Notebook). **Mutable** (controller edits it in place).
- `CanvasPage` (`canvas_page.dart`): one sheet; size, `PageBackground` (color + `BgPattern` blank/ruled/grid/dotted), optional `PdfSource {assetId, pageIndex}` (= PDF-backed page), `elements` list (z-order = list order).
- `CanvasElement` (`element.dart`): **sealed, deliberately mutable** — `StrokeElement` (tool pen/highlighter, color, size, points with pressure `p`, transient `cachedOutline` Path invalidated on mutation), `TextElement` (rect, rotation, `runs: List<TextRun>` for per-range styling — see Text below — plus baseline fontFamily/size/color/bold/italic/align as fallback and toolbar default), `ImageElement` (rect, rotation, assetId), `AttachmentElement` (rect, rotation, assetId, name, mime — a visible "attached file" chip; "Add as attachment" drops one on the current page, tapping it in lasso/text mode opens the file via `open_filex`; export draws the chip, embeds the file as a PDF document attachment, and overlays an invisible `PdfActionAnnotation` + `PdfJavaScriptAction` running `exportDataObject({cName, nLaunch:2})` so clicking the chip in Acrobat/Foxit-class readers opens the embedded file — embed names are de-duplicated per assetId since `exportDataObject` looks up by name; simpler viewers still reach it via the attachments panel). All support `deepCopy({withNewId})`, `translate`, `scaleBy` (uniform; no-op for `TextElement`'s size — see Text), `rotateBy`. Ids from `newModelId()`. All carry a `zIndex` (json `zi`, default 0): strokes and objects live in separate lists (sync model), so painter **and** exporter draw `zOrderedElements(page)` — the combined list stable-sorted by zIndex (ties keep strokes-under-objects) — letting Send-to-back put an image *behind* ink; `_reorderSelection` sets `max+1`/`min-1` and `_stamp`s the elements so the z change syncs. **Because the tie-break renders objects *over* strokes at equal z, a plain-inserted image lands on top of ink** — so freshly added/pasted images go through `CanvasController.addImageBelowInk`, which sets the image's z just below the lowest stroke z (default 0 → image at −1) so ink (now or drawn later) stays on top while the image still sits above the page background; images tie among themselves and fall back to insertion order (newest on top). Text/attachments still insert on top via `addElement`. **Image paste is caret-aware** (`_placeImage` in `canvas_screen`, 07/17/26): with the caret in a text box, an image lands via `CanvasController.insertImageAtCaret` as a **block** — never inline — on the line *after* the caret, left-aligned to the box, with the text following the caret pushed below it (three shapes: caret at end → nothing moves; at start → image takes the box's top-left and the box slides under; middle → the box truncates and the remainder becomes a **new** box under the image). One undoable op; `_stamp`s the truncated box and `_belowInkZ`s the image. It **commits the live session first** — the caret indexes the box's *committed* runs, and the editor's controller holds the truth until then — and re-resolves the element **by id** afterwards, since committing routes through `updateTextElement`→`_replaceElements`, which swaps in a `deepCopy`. The same-page boxes deliberately get **no linkId**: the linked-text chain is page-ordered (`insertTypingContinuation` looks for the next part on the NEXT page), so a same-page part would make the chain ambiguous — known limit: splitting an already-flowed box leaves the after-text outside that chain. **Text that no longer fits under the image flows onto the next page** (fixed 07/17/26 — it used to be placed below the page edge, where the painter's per-page clip made it *invisible*, and nothing ever reflowed it: the typing-overflow check only fires from `_remeasureEditing` for the box being **edited**, and a box this op just placed isn't). `_fitRunsInHeight` splits the after-text at the room left under the image (returning *nothing* fitting when there isn't even one line's worth — `splitRunsByHeight` always keeps one line in its first chunk, which would strand exactly that line), and the remainder goes through **`_buildContinuation`** — `insertTypingContinuation`'s body refactored to *return* its op instead of running it, so the paste **folds it in and stays ONE undo** (that build already mutates, so the op it returns must be `_doOp`'d exactly once). The overflow links from the last part still on the page (the below-box, else the truncated box above the image) — cross-page, so the chain stays page-ordered. A **tall image itself** is still clipped at the page edge rather than moved (accepted). With **no** caret the image lands at the **top-left of the visible viewport** (clamped onto the page), not the page centre as before. Triggers: **Ctrl/Cmd+V and the context-menu Paste while editing** both route through an `Actions` override of **`PasteTextIntent`** on the editor's `TextField` — necessary because `_onCanvasKey` bails out entirely while a text edit is open (the field owns its keys) and Flutter's default paste reads `text/plain` only, so an image on the clipboard was silently dropped; text still takes the normal path (pasted through the controller, since `TextField` exposes no handle on its inner `EditableText` to delegate `pasteText` back to). The **Add sheet's Paste** works too via `_lastCaret`, the caret remembered at commit time — that button lives outside the field, so its tap fires `onTapOutside` and commits the session *before* the handler runs; `_lastCaret` holds an **id** (not a ref, for the `deepCopy` reason above) and is only honoured while the text tool is still active. When adding a subtype: include `zi` in toJson/fromJson/deepCopy or layering silently resets on reload. Adding a subtype breaks sealed-switch exhaustiveness in `canvas_painter`, `canvas_controller` (`_lassoHits`, `applyColorToSelection`), and `pdf_exporter` — the compiler flags every site.

### Persistence (`lib/services/notebook_service.dart`)

```
<appsupport>/notebooks.json                                        # notebookId -> Notebook
<appsupport>/notebooks/<nbId>/sections/<sectionId>/section.json    # structure only (rows, defaults, attachments)
<appsupport>/notebooks/<nbId>/sections/<sectionId>/pages/<pageId>.json   # one CanvasPage per file
<appsupport>/notebooks/<nbId>/sections/<sectionId>/assets/<sha256>.<ext> # content-addressed PDFs/images (dedup)
```

`<appsupport>` is `getApplicationSupportDirectory()` — the OS **application-support** dir, deliberately **not** `getApplicationDocumentsDirectory()`: on desktop the latter maps to the user's real `~/Documents` and dumped the whole store (JSON + `settings.json` + `sync_journal.json` + `drive_index_*.json`) loose in there. App id is `io.github.ravinduRepo.omininote` on every platform (Android `applicationId`/`namespace`, macOS `PRODUCT_BUNDLE_IDENTIFIER` + entitlements keychain group, Linux CMake `APPLICATION_ID`; Windows has no bundle id so `Runner.rc` sets `CompanyName=io.github.ravinduRepo` + `ProductName=omininote`). `<appsupport>` resolves to: Windows `%APPDATA%\Roaming\io.github.ravinduRepo\omininote`; macOS `~/Library/Containers/io.github.ravinduRepo.omininote/Data/Library/Application Support/io.github.ravinduRepo.omininote` (sandboxed); Linux `~/.local/share/io.github.ravinduRepo.omininote`; Android `/data/user/0/io.github.ravinduRepo.omininote/files` (private internal). All three data-store call sites use it — `NotebookService.init`, `SettingsService.init`, `DriveService._loadIndex/_saveIndex`; only `notebook_share_ui.dart`'s export staging stays on `getTemporaryDirectory()`. iOS is still `com.example.omininote` (not a target). No data migration exists — this changed the base dir with no in-app move, so a pre-change install's old `~/Documents` store is orphaned (acceptable: pre-release, no real users).

All writes are **atomic** (`_writeAtomic`: tmp file + delete + rename). Editing a page rewrites only its file; structural changes rewrite `section.json`. **Page-JSON is (de)serialized off the main isolate (07/13/26, perf):** `jsonEncode`/`jsonDecode` of a page's full stroke/object payload is pure CPU that used to jank drawing (the debounced autosave) and canvas-open (decoding every page). `savePage` now offloads the encode via `Isolate.run` **only for dense pages** (`strokes+objects ≥ _kPageOffloadElements`=120) and **never for tombstone/purge saves** (they stay inline so a delete-save can't lose a write-ordering race to an in-flight pre-delete autosave — `CanvasController.deletePage` fires its save via `unawaited`); light pages encode inline (the isolate spawn+copy would cost more than the work). `loadPages` reads page files in bounded chunks (`_kPageReadChunk`=32, so peak memory doesn't hold the whole canvas) and batch-decodes each chunk via `Isolate.run` when it's heavy (`_kPageDecodeChars`=256KB), else inline — then builds the models + prunes tombstones on the main isolate (`fromJson` touches ui/model objects that can't cross the boundary). `toJson` also stays on-main (it walks live elements whose transient `cachedOutline` `Path` isn't sendable); only the resulting pure-data map is shipped. The decode helper `_decodePageJsonsSync` is **static** so its `Isolate.run` closure can't drag `this` (File handles, caches) across the boundary. Both offloads fall back to inline on any isolate failure — a hiccup must never drop a save/load. **New items land below the selection (07/20/26):** `createSection`/`createCanvas`/`createCanvasFromPdf` take an optional `afterSectionId`/`afterCanvasId`; the desktop shell's header "+" passes the currently-selected section/canvas id so a new item is inserted **right after it** (`TreeOps.insertLeafAfter`, which finds the target anywhere in the tree — top level or inside a folder — and splices in after it), falling back to end-of-list when nothing's selected or when adding *into* a folder (folderId set). Mobile has no persistent list selection, so it keeps appending. The notebook's whole contents tree (sections, super-sections, order, colors, collapse state) lives in `notebooks.json` — `saveNotebook(notebook)` persists it; `reorderNotebooks` rewrites the index object in the new key order (JSON object order is preserved through decode/encode). Group ops are tree-walk helpers (`_findGroup`/`_removeSectionNode`/`_spliceOutGroup` for ungroup/`_removeGroupNode`); `deleteGroup` also deletes its contained sections' dirs, `ungroup` splices a group's children into its own position (keeping sections). A legacy pre-canvas `notebooks.json` (detected by `pageIds` keys) is moved to `notebooks_legacy.json` and the app starts fresh (spec §16). Page files orphaned by deleted pages are left on disk (undo may restore them; harmless). `RenderCache` (`render_cache.dart`) renders PDF pages via pdfrx engine (`PdfDocument.openFile` → `page.render` → `createImage`) into an LRU of `ui.Image`s bucketed by zoom, plus decodes raster image assets; `pdfrxFlutterInitialize()` is called in `main()`. **Container maps read bounded-concurrently (07/14/26, perf):** `getSectionMap`/`getCanvasMap` fan their per-section/per-canvas file reads through `_mapBounded` (8 in flight — a self-contained twin of `SearchService._mapBounded`, kept here so the service depends on nothing above it) instead of a serial `await`-in-`for`, so opening/switching a section with many canvases (or a notebook with many sections) no longer chains N disk round-trips; the result is a lookup map, so read-completion order is irrelevant. `getSectionMap(id, {notebook})` also takes an optional preloaded `notebook` so callers iterating already-loaded notebooks (the desktop shell's `_loadAll`/`_reloadNotebook`) skip re-decoding the whole `notebooks.json` once per notebook.

**Note:** the on-disk shape now has a **Canvas** level between Section and pages — `notebooks/<nb>/sections/<sec>/canvases/<cid>/canvas.json` (rows/defaults/attachments) + `.../canvases/<cid>/pages/<pid>.json` + `.../canvases/<cid>/assets/<sha>.<ext>`. Section is a tree of canvases; Canvas owns the rows. All five model types (Notebook/Section/Canvas/CanvasPage/CanvasElement) carry a sync **envelope** (`schemaVersion,id,rev,updatedAt,deviceId,deletedAt`); save methods call `bumpRev(deviceId)` so `rev`/`updatedAt` actually advance (they're the LWW signal). `CanvasPage` uses the **stroke-set model** (`strokes` + `erased` tombstones + `objects`), not a flat `elements` list.

### Cloud sync (`lib/services/sync_service.dart`, `drive_service.dart`, `auth_service.dart`, `desktop_oauth.dart`, `sync/merge_engine.dart`)

Two-way Google Drive sync (spec: `implementation_plan.md`, status: `task.md`). The Drive tree **mirrors** local storage under a visible `omininote/` My-Drive folder (scope `drive.file`). **Auth:** Android via `google_sign_in` (silent restore); desktop via loopback **PKCE** OAuth (`desktop_oauth.dart`) with the `refresh_token` in `flutter_secure_storage` and the ~1h access token refreshed proactively (5 min pre-expiry) + reactively — `_AuthedClient` re-fetches headers per request so the fresh token is always used. **`DriveService`** keeps a persisted `drive_index.json` (relPath ↔ fileId ↔ `headRevisionId`) so updates are O(1) and our own uploads are echo-suppressed; `listAllFiles` does a single paginated `files.list` and rebuilds relative paths from parents (treating every "omininote" folder as a root → duplicate-root create/create races self-heal), and **excludes the `omininote/shared/` folder** (public "Share a link" `.omninote` ZIP bundles — not synced store; pulling one as JSON blew up `utf8.decode`). **Both pull entry points gate on `NotebookService.isSyncedRelPath`** (`notebooks.json` / `notebooks/…`): the incremental change path always did; the full-resync loop now does too, and wraps each `_pullFile` in a try/catch so one unreadable file can't abort the whole resync. **`MergeEngine`** is pure/unit-tested (`test/merge_engine_test.dart`): `notebooks.json` = union map by notebook id + per-id LWW (so two devices' notebooks both survive first sync); `section.json`/`canvas.json` = **structure-aware merge** (07/18/26): LWW by `(rev,updatedAt,deviceId)` for every scalar field, but the *membership* is unioned — a section's canvas leaves, a canvas's page rows + bookmarks + attachments — so a canvas/page added concurrently on two devices is never dropped by a whole-doc clobber (`reconcile` routes `/section.json`→`_foldSectionMembership`, `/canvas.json`→`_foldCanvasMembership` via `_mergeStructuredDoc`; loser's orphans appended — leaves top-level, pages as new rows with a **stable derived id `r-<pageId>`** so both devices produce byte-identical output; the merge never bumps `rev`, so it converges in one round with no push loop). Deletion is unaffected (item-level `deletedAt` tombstones remain the durable gate; the UI filters leaves whose item is tombstoned, so a unioned-in dead leaf is invisible, never a resurrection). Only arrangement (order + folder grouping) stays LWW. No schema change/migration. page files = set-union of immutable strokes + grow-only erase tombstones, LWW for text/image objects filtered by a grow-only **object-delete tombstone** set (`CanvasPage.deletedObjects`), plus background. **Deletion is always soft-delete, never a hard remove/splice** — this is the load-bearing rule that makes deletes durable (v3 fix, see KNOWN_ISSUES "Fixed"): `deleteNotebook`/`deleteSection`/`deleteCanvas` set `deletedAt`+bump `rev` and keep the entry/file on disk (a hard-removed entry has no envelope left to win a merge comparison, so a live remote copy resurrects it — this was the actual bug, not a design gap); `getNotebooks`/`getNotebook`/`getSection`/`getCanvas` filter `deletedAt != null` so the UI still sees it as gone. Same idea one level down: `CanvasController.deleteSelection`/`removeElement`/`_addElementsOp.revert` tombstone (add to `erased`/`deletedObjects`) *and* physically remove from `strokes`/`objects` in the same op — the physical remove is for rendering (painter/hit-test read `page.strokes`/`page.objects` directly, no tombstone filter at that layer), the tombstone is what travels through merge. Page deletion (`CanvasController.deletePage`) explicitly calls `savePage` with `deletedAt` set before the page leaves the in-memory `pages` map (the normal debounced autosave keys off that map and would otherwise never persist the tombstone); `NotebookService.loadPages` prunes any row still referencing a tombstoned page (self-heals a canvas.json structural-LWW race) and guarantees a canvas is never left with zero pages. Soft-deleted notebooks/sections/canvases **and individually-deleted pages** are visible in the **Recycle bin** (Settings → Deleted items → `bin_screen.dart`): `NotebookService.listDeletedItems` reads tombstoned files directly (the normal getters filter them; purged markers are hidden); `restoreBinItem` clears `deletedAt` + bumps `rev` (the restore then *beats the tombstone in LWW on every device* — sync propagates it automatically) and re-links the tree leaf its deletion removed. Restore is disabled while the item's parent is itself deleted (`parentAlive`). **Pages are a 4th bin level** (`BinItemType.page`; `listDeletedItems` scans `pages/*.json` under *alive* canvases only — a deleted canvas's pages are subsumed by the canvas entry): `restorePage` clears the tombstone and re-links the page as a **new row appended at the bottom** of the canvas (original position is unrecoverable after restructuring — same simplification as sections/canvases restoring to end-of-tree). **Open-canvas restore is dispatched** — `NotebookService.restorePage` first calls `SyncService.restorePageInOpenCanvas` (keyed by the `CanvasSyncListener._canvasListeners` registry); if the canvas is open, the live `CanvasController.restorePage` does the row-append + tombstone-clear **in memory + `await flushSaves()`** (a disk-level restore would be clobbered by the controller's own autosave), else the disk path (`getCanvas` → `savePage` clears + bumps → append row → `saveCanvas`) runs. A page's terminal purge (`purgePage`, and the 90-day `_gcPagesUnder` now routes here too — was a local-only delete that leaked the file on Drive) strips content + stamps `CanvasPage.purgedAt` in place; the page file **is** the marker (a page has no subtree), so it needs no folder-delete / `applyPagePurgeLocally` / `isPurgedContentPath` branch — `MergeEngine.mergePage` propagates it (terminal, grow-only, earliest-wins, content stripped). **Super-sections (folders) are a 5th bin level** (`BinItemType.folder`): deleting a super-section moves its whole `FolderNode` subtree into `Notebook.deletedFolders` / `Section.deletedFolders` (`DeletedFolder {node, deletedAt, purgedAt?}` in `tree.dart`) — the contained items' files stay on disk *hidden* (the tree no longer references them, so they don't double-list). `restoreFolder` re-appends the subtree; `purgeFolder` drops the record + purges every contained leaf; synced via the notebook/section LWW (no merge change). Folder delete is confirm-less + animates (`ItemTreeView._deleteFolder`). **Restore-across-devices caveat** (same as the 3 container levels): a restore's row-append can lose to a concurrent higher-rev `canvas.json` (page alive-but-unreferenced there until touched). Restoring a page while its canvas is open leaves the original delete op on that session's undo stack — `deletePage`'s `revert` guards against re-inserting a row for a page that was already re-linked out of band (so undo can't create a duplicate-row corruption); a subsequent redo self-heals on reload. **Purge (stage 2, permanent — bin "Delete forever" / "Empty bin" / the 30-day auto-GC, one mechanism):** upgrades the tombstone to a **terminal `purgedAt` marker** — content subtree hard-deleted from local disk *and* Drive (`DriveService.deleteFolder`, queued persistently in `sync_journal.json`'s `purges` list, drained with retry in `_flushPush` after file uploads so markers land first; idempotent — 404 means another device already did it. **`deleteFolder` is best-effort per-item (07/15/26):** a cascading `files.delete(folderId)` 403s under the `drive.file` scope ("not granted write access to all of the children"), so it walks the subtree (`_deleteSubtree`) and deletes each app-created file/subfolder **individually** bottom-up (`_deleteOne` swallows 404 *and* 403 — a non-owned leftover we can't touch is skipped, not fatal, since the purge is enforced by the marker + content-path filter, not by this delete). It returns normally after best-effort, so `_drainRemotePurges` **removes the entry from the queue** — fixing the previous infinite-retry+log-spam where a 403 rethrew and stayed queued forever; network errors still propagate so offline stays queued), while the tiny marker doc survives *forever* (the `notebooks.json` entry / a stripped `section.json`/`canvas.json` — never GC'd; that permanence is what stops a stale offline device's live copy from resurrecting the item). `purgedAt` is **grow-only in merges** (`MergeEngine._withPurgeOverride` — exempt from LWW, earliest wins, forces `deletedAt`), so a restore racing a purge loses deterministically. Purge propagates like any edit: devices pulling a marker run `SyncService._applyPurgeMarkers` → `NotebookService.applyXPurgeLocally` (wipe local content, record the prefix, re-enforce the Drive delete). Push/pull both filter content under purged prefixes via `NotebookService.isPurgedContentPath` + the device-local `purged_index.json` cache (rebuilt each GC walk) — the marker files themselves must keep flowing (they carry the purge), which the filter exempts. Local-only notebooks purge as a plain local delete. Consequence (documented): offline edits to a since-purged item are dropped on reconnect. `NotebookService.runGarbageCollection` (on launch + after every `fullResync`) routes expired tombstones through this same purge path and trims stale page-level tombstone entries. **The GC scan runs off the main isolate (07/15/26):** it decodes *every page file in the store*, which janked launch — so the read/decode walk is static `_gcScan` (plain-data in/out, run via `Isolate.run` with inline fallback) returning a `_GcActions` list of slash-joined id paths; the mutations (purges, dir deletes, `_addPurgedPath`, Drive queue) run on-main afterwards. The scan never descends under a purged/expired parent, so the action lists can't overlap. Tombstone-list compaction rewrites stay inside the scan (they deliberately bypass `onLocalFileSaved` either way — local-only trims). **`SyncService`** orchestrates: `NotebookService._writeAtomic` → `onLocalFileSaved` marks the relPath dirty in a **persistent journal** (`sync_journal.json`, survives kill) → debounced (1.5 s) sequential upload with backoff. Pull = Changes API poll every 15 s: known fileIds are downloaded+merged in place with echo suppression; an **unknown** fileId (a new file from another device) is placed by `DriveService.resolveFileId` (parent-chain walk, folder-cache-accelerated; records the mapping with **no** head so echo suppression can't skip the first download) and pulled inline — only a failed resolve or a 410-Gone falls back to a debounced **`fullResync`** — the one routine used for first-sign-in **bootstrap**, 410 recovery, and the Settings **"Repair sync"** button (`_onSignedIn` only bootstraps when there's no changes token / empty index, else resumes incremental). `fullResync` takes its changes baseline token **before** `listAllFiles` (after = changes pushed during a slow resync are skipped forever), and `_pullFile` skips downloads whose `headRevisionId` matches the last reconciled head — `listAllFiles` must **preserve** previously-reconciled heads instead of stamping current ones, or the resync would skip every file. **Sync-correctness invariants for canvas edits:** every element-mutating op commit must `_stamp` (bump `rev`/`updatedAt`/`deviceId` on) the affected elements or LWW ties and devices never converge; every path that removes an element from a page's arrays must write an `erased`/`deletedObjects` tombstone in the same op (the eraser's old first-apply replay guard skipped exactly this and erased ink resurrected). **Stroke/object deletion is rev-based LWW** (07/12/26 — like the bin's notebook/section/canvas deletion): `EraseTombstone` records the element's `rev` at delete time, and `MergeEngine.mergePage`'s filter keeps an element dead only while `element.rev <= tombstone.rev` (was a binary grow-only `!containsKey`). So **undo/redo never removes a tombstone** (grow-only *storage*; un-tombstoning locally would lose to a device that already pulled it): instead, delete = `_tombstoneFor` (tombstone at `el.rev`), revive = keep the same id and `_bumpAliveOn` (bump rev above the tombstone). rev climbs monotonically across undo↔redo so each action out-revs the last. Same-id revival keeps the **undo stack intact** (earlier ops still find the element by id — the fresh-id approach broke this). Shared helpers `_tombstoneFor`/`_bumpAliveOn`/`_tombstoneSlots`/`_reviveSlots` (+ the `_ElSlot` holder) back every delete path: `deleteSelection`, `removeElement`, `_addElementsOp` (insert/draw/paste/image), `_commitErase` (whole + partial eraser), `deleteLinkedText`, split-paste/`insertTypingContinuation` reverts, and **`_reparentSelection`** (cross-page move = delete-on-source + add-on-target, same ids; the landed copy is bumped above any tombstone on its page, which is what lets a move-back out-rev the stale origin tombstone). **Semantic:** a concurrent higher-rev remote edit resurrects a deleted element (edit-wins-over-delete, same as the bin); a passive copy at/below the tombstone rev stays deleted. Covered by `merge_engine_test` (rev-based deletion group + partial-erase-across-sync), `canvas_controller_test` (undo-chain), `partial_eraser_test`, `split_paste_test`. Writes land on disk via `writeAtomicPublic`/`writeAtomicBytesPublic` (no re-upload loop) and bump `SyncService.dataVersion`, which the home + desktop shells listen to and reload. **Open-canvas live merge:** `CanvasController` registers a `CanvasSyncListener` (keyed by canvas id) in its constructor; `_pullFile` routes pulled page files to `applyRemotePage` (set-union merge into the *same* in-memory `CanvasPage` instance — painter/undo refs stay valid — dropping remotely-tombstoned pages, marking the controller dirty **only when the union holds something Drive lacks**, else it would echo revisions forever) and pulled `canvas.json` to `applyRemoteStructure` (skipped while local structural edits are unflushed). Without this, an open canvas's autosave clobbered every pulled merge with stale in-memory state → the delete-resurrection ping-pong. Perf: `DriveService._folderIds` caches folder path→id (warmed during `listAllFiles`) so uploads don't re-resolve their folder chain with per-segment `files.list` calls. **Merges run off the main isolate for large files (07/15/26):** `_pullFile` routes both merge branches (`MergeEngine.reconcile`, and the scoped `notebooks.json` merge — now static pure `_mergeIndexScoped`, its main-isolate inputs `defaultId`/`localOnlyNotebookIds` captured as plain data) through static `_runMerge` — inline below 256KB combined JSON (`_kMergeOffloadChars`, mirroring the page-decode gate), `Isolate.run` above it, inline fallback on any isolate failure (a hiccup must never drop a merge; only the string/bool `MergeResult` crosses the boundary — heavy merges were blocking frames while drawing during sync). Orchestration (echo suppression, `writeAtomicPublic`, purge markers, the open-canvas live merge) stays on-main. **Phase 2 trims (07/15/26, same jank):** `_notifyOpenCanvas` (which fires exactly when the pulled canvas is open under the user's pen) offloads its dense-page `jsonDecode` via `_decodeJsonMap` (same 256KB gate + inline fallback; `CanvasPage.fromJson` + the live merge stay on-main); the push path uploads file **bytes** straight through (`DriveService.uploadJsonBytes` — `readAsString`+`uploadJson` was a full utf8 decode+re-encode of every dense page on main); `downloadJsonById` offloads its utf8 decode ≥256KB (`_kDecodeOffloadBytes`). `main.dart`'s `_NoteAppState` (a `WidgetsBindingObserver`) flushes pending on background and syncs on resume. **Gotcha:** the sync-write path must NOT go through `savePage`/`saveNotebook` (those `bumpRev` and re-trigger `onLocalFileSaved` → an upload echo loop) — always use the `*Public` shims. See KNOWN_ISSUES "Cloud sync" for remaining limitations (open-canvas live merge, structural *ordering/folder-grouping* still LWW — membership is now unioned, no Conflicts Inbox, purge-drops-offline-edits, Google Cloud setup prerequisites).

### Canvas (`lib/canvas/` + `lib/screens/canvas_screen.dart`) — the core

- `canvas_layout.dart`: pure layout — rows stacked vertically, pages left-aligned in rows, `kPageGap`=**0** (pages sit flush for a clean continuous document; a hairline border separates them); produces `PageLayout` rects in canvas space. Inserted PDF pages are normalized to the section's default width (aspect ratio preserved) so the column is uniform-width.
- `canvas_controller.dart` (`CanvasController extends ChangeNotifier`): **the app owns the viewport** (`zoom` + `pan`, exposed as `viewportMatrix`; pdfrx has no camera role). **Smooth scrolling** is ticker-driven: touch drag pans 1:1 (`panImmediate`) then flings on release (`flingBy`, velocity + exp decay); the wheel glides toward a target (`scrollByWheel`, exp smoothing); `stopScrollAnimation()` halts momentum on re-grab. **Scrollbars**: `verticalScrollbarThumb`/`horizontalScrollbarThumb` geometry drawn by the painter; mouse can drag the thumb (`beginScrollbarDrag`/`updateScrollbarDrag`), touch is indicator-only (finger pans directly). Also holds tool state, gesture routing (`startToolGesture/update/end` → page-local, `_activeGestureTool` locks the tool for the gesture), **cross-page freehand** (07/20/26 — a pen/highlighter stroke drawn past its page edge is no longer clamped to the origin page: `updateToolGesture` keeps the raw unclamped point, and `endToolGesture` routes freehand through `_commitStrokeAcrossPages`, which walks the polyline once and — only if it actually crosses — splits it into a fresh `StrokeElement` per page it touches, inserting a shared edge point per transition (`_exitPoint`) so the halves meet flush (pages sit gap-0), committed as one multi-page undoable op (single-page strokes keep the original fast path + id). Cost is one O(points) scan on lift; the painter draws the *live* active stroke **outside** the per-page clip, translated by the origin page's top-left, so the crossing previews seamlessly. A run over empty canvas / a gap is dropped. Covered by `canvas_controller_test`), eraser (one undo op per gesture; **Stroke** mode removes whole strokes, **Partial** mode — `eraserPartial`, with adjustable `eraserSize`, both persisted device-local via `SettingsService.setEraserPrefs` — splits a stroke at the erased gap: original tombstoned, surviving point-runs become fresh strokes with the original's style/z, so the erase syncs as tombstone + union like everything else; undo tombstones the segments; a segment created mid-gesture re-splits without tombstoning since it was never persisted; **perf (07/14/26):** each pointer move bbox-rejects strokes via the cached `StrokeElement.bounds` before the per-segment hit-scan, and the page's committed ink is painted **directly** during the gesture instead of re-recording the whole page picture on every erased stroke — see painter `isErasingPage`), lasso selection (point-in-polygon; live refs + `selectionBounds`), selection drags (move/uniform-resize/rotate, deep-copied before/after; **dropping a moved selection over another page reparents the elements to it** — `_reparentSelection` converts coords to the target's local space, one op over both pages), internal static clipboard, **op-based undo/redo** (`_CanvasOp{apply, revert}` over deep copies, capped 100), page/row structure ops, attachments, over-scroll page-add (right → horizontal page, bottom → append; `overscrollThreshold`=200px deliberately large so a normal scroll that reaches the edge never adds by accident, and the painter's "+" hint stays hidden until the pull passes `overscrollHintFloor`=90px so it doesn't flash during ordinary edge-scrolling; Local space here is the box's top-left and **ignores rotation**, matching the neighbouring link/checkbox hit-tests and `_textElementAt`. Editing UI is a positioned `TextField(controller: RichTextController)` overlay committing on tap-outside; the painter skips `editingElementId`. **PDF export is per-run:** `placedRunFragments` (`text_measure.dart`) lays the box out with the *same* `TextPainter` the painter uses and splits it into per-line styled fragments with positions (mixed sizes baseline-align via each fragment's own selection-box top); `pdf_exporter.dart`'s `_drawText` draws each fragment with its own `PdfStandardFont`/color at that position (+0.15 em top nudge to match the painter's 1.3 line height), so line breaks, alignment, and styling match the screen exactly.

### PDF export (`lib/services/pdf_exporter.dart`)

`PdfExporter` interface; `SyncfusionPdfExporter` implements spec §10: one Row → one PDF page; multi-page rows merge **flush** into one wide page sized exactly to the summed page widths (see Gotchas — page size must be assigned via a fresh `PdfPageSettings`, never by mutating `.size`); PDF-backed pages draw the original imported page via `createTemplate()`/`drawPdfTemplate` (**vector — text stays selectable**); strokes exported as filled vector paths using the *same* perfect_freehand options as the painter, and **smoothed to match** (`_smoothStrokeOutline` samples the same quadratic-midpoint curve as `CanvasPainter._smoothOutlinePath` into a dense polygon — `PdfPath` has no quadratic primitive, so densify rather than bézier); text as real PDF text (standard fonts Helvetica/Times/Courier); images embedded; highlighter via `setTransparency(0.35)`. Background **pattern** (ruled/grid/dotted) is drawn as vector lines/dots on blank pages (`_drawPattern`, mirroring `CanvasPainter._paintPattern`'s exact 26pt-spacing geometry) — PDF-backed pages skip it since the imported page image covers the whole page, same as on screen. Saved via `FilePicker.saveFile` (desktop returns a path without writing — the screen writes bytes if needed). **Off the main isolate (perf 07/14/26):** the Syncfusion assembly + stroke-outline math is heavy pure CPU that froze the UI, so `exportPdfInIsolate` (`pdf_export_isolate.dart`) runs it in a spawned isolate. The boundary carries only plain data — each canvas as `toJson` + referenced asset **bytes** + **precomputed text fragments** (`placedRunFragments` uses `TextPainter`, which needs the Flutter engine and so **can't run in a background isolate** — it's computed on the main isolate and passed via `PdfExportItem.placedText`, which the exporter's `_drawText` uses instead of laying out). Per-canvas progress streams back over a `SendPort` (`exportTree` gained an `onProgress(done,total)` callback); any isolate failure falls back to an on-main export (never fail). `runTreeExport` (notebook/section) and the single-canvas `_exportPdf` now show a **non-modal `ProgressBanner`** (`lib/utils/progress_banner.dart`, a live `MaterialBanner`) instead of a blocking dialog, so the app stays usable while exporting. The export **collection** (`collectNotebook/SectionExportItems`) also loads canvases + pages bounded-concurrently (`_loadExportLeaves` → `_mapBounded`, tree order preserved). Covered by `test/pdf_export_isolate_test.dart` (isolate output reopens to the same page count/size + outline as the on-main export). **Notebook bundle** send-a-copy/import (`notebook_bundle_service.dart`) is off-main **and streamed to/from disk** (the OOM fix): both directions move **one zip entry at a time between disk and a spawned isolate**, so a gigabyte notebook never sits fully in RAM and only file *paths* + the tiny manifest/index blobs cross the isolate boundary. **Export** (`exportBundle`) uses `ZipFileEncoder` and returns a **temp-file path** (callers move/copy/read the file, delete when done). **Import** (`importBundleFile(zipPath)`) uses `InputFileStream`→`ZipDecoder().decodeStream`→`OutputFileStream`, validating manifest+index from the central directory **before** writing any `data/…` entry into the staging notebook dir, so a bad file leaves nothing on disk; on mid-extraction failure the partial staging dir is removed. The byte-taking `importBundle(bytes)` is kept only for sources that can't hand a path (a downloaded share-link body) — it spools the bytes to a temp file then calls `importBundleFile`. The file-picker import sets `withData:false` and streams from the picked path (loading bytes there defeated the fix); `main.dart`'s open-with routes the incoming path straight to `importBundleFileUi` (no `readAsBytes`). Import falls back to inline extraction only if the isolate can't be *spawned* (nothing written yet); a running isolate's error is a real failure and isn't retried. Progress via `ProgressOverlay` (`onProgress(fraction,label)`), inline fallback on export isolate failure. Covered by `test/notebook_bundle_import_test.dart` (`debugStreamUnzip` round-trips a `ZipFileEncoder`-built bundle losslessly, rejects a non-Omininote/corrupt zip without writing, and honors the zip-slip guard).

### Voice recording & playback (audio-sync) (07/18/26)

Record audio over a canvas, play it back, and see the ink glow in time with playback. **Model:** `AudioRecording {id, name, assetId, startedAt, durationMs, createdAt}` (`models/canvas.dart`) lives in `Canvas.recordings` — rides canvas.json (synced structurally), **unioned by id** in `MergeEngine._foldCanvasMembership` (so a take recorded concurrently on two devices both survive), carried through every canvas-rebuild site (relocate/copy in `notebook_service`, `applyRemoteStructure`), and cleared in the purge-strip. The `recordings` JSON key is **omitted when empty** so old canvas.json stays byte-stable. The audio is a content-addressed asset (`NotebookService.putAsset`, `.m4a`), synced like any image (no merge-engine change). Both "Record audio" and "Recordings" are in `kAddActionSpecs` (`toolbar_actions.dart`) — they're **content additions, so they live in the "+" (Add) menu**, and are promotable to the bar via Customize toolbar like any other add action (gated by `shown(id)`, dispatched by `_buildPromotedButton`; the `record_audio` bar button wraps itself in a narrow `ValueListenableBuilder` on `isRecordingAudioNotifier` — turning red while recording — rather than the static cluster, per the "smallest-possible listener" rule). **Capture** (`services/audio_recorder_service.dart`, `record` plugin, aacLc, permission-gated): the "Record audio" entry starts it; a floating `_RecordingBar` (pulsing dot, live timer, Stop/discard) overlays the canvas in **both** normal + full-screen layouts (it's in the shared `_buildCanvasArea` Stack) so you draw *while* recording. `CanvasController.startAudioRecording`/`stopAudioRecording` (stores the asset + appends the recording, structural save + flush-then-reindex like bookmarks)/`cancelAudioRecording`. **Playback** (`services/audio_playback_service.dart`, `audioplayers`): the "Recordings" entry opens the **`_AudioPlayerBar`** — a small non-distracting floating player (bottom-center of the shared canvas Stack, so it shows in normal + full-screen) with a **recording picker** (`PopupMenuButton` whose rows carry inline rename/delete; the record `({op, rec})` result routes play/rename/delete), a scrubber, quick **±10s skips** (`AudioPlaybackService.skip`), play/pause, and a cycling **speed chip** (`speed`/`setSpeed`, re-applied on each `play()` since a fresh play resets the rate). `_audioPlayerOpen` (canvas-screen state) gates it; opening with no recordings just toasts; closing stops playback. This replaced the old Recordings bottom sheet + `_RecordingTile`. Both services are **lazily** created per controller (a canvas with no recordings never touches the plugins) and disposed with it. **Replay after finish:** `onPlayerComplete` resets `position` to zero and clears the paused flag, and `play()` only `resume()`s when it's the *paused* current take — otherwise (different take, stopped, **or finished**) it restarts from the top, so a finished recording replays. A `_resumeFrom` set by scrubbing while stopped lets you drag the playhead back and Play from there (`play()` seeks to it after a fresh start). Without this, `play()` blindly `resume()`d an ended player → couldn't replay. **Audio-sync glow** (`lib/utils/audio_sync.dart`, pure + `test/audio_sync_test.dart`): the key trick is that `StrokeElement` **already carries `createdAt`** (wall-clock), so no per-stroke schema change — `CanvasController.audioPlayheadNotifier` (`ValueNotifier<DateTime?>`, updated from the playback position; null unless actively playing) holds `recording.startedAt + position`, and `CanvasPainter` **merges it into its repaint listenable** (`Listenable.merge([controller, audioPlayheadNotifier])`) so playback repaints only the canvas, drawing a soft accent halo (`_paintAudioGlow`, reusing each stroke's cached outline) on strokes whose `createdAt` is within `kAudioGlowWindow` of the playhead — **outside** the picture cache, a pure no-op when nothing plays. **Playback auto-follows the glow (07/19/26):** `_updateAudioPlayhead` → `CanvasController._followAudioGlow` pans (`ensureCanvasRectVisible` — keep zoom, move only when off-screen, mirroring read-aloud's `_followReading`) to the union of glowing strokes on the newest glowing stroke's page; always-on (no toggle), runs only from the playback position listener (~200ms ticks) so it costs nothing during normal drawing/typing/panning. Deps: `record` + `audioplayers` (both resolve cleanly, full Android/desktop coverage); Android `RECORD_AUDIO`, macOS `device.audio-input` entitlement + `NSMicrophoneUsageDescription`. **Not yet device-verified** (no mic in the build session); see KNOWN_ISSUES for that + the Linux-backend / discard-on-close / PDF-export / clock-skew caveats.

### Read-aloud / text-to-speech (07/19/26, not yet device-verified)

Speak the canvas's text: typed `TextElement`s **and** imported-PDF text. **Future-proof extraction layer** (`lib/utils/readable_text.dart`, pure + `test/readable_text_test.dart`): the whole pipeline's currency is `ReadableSpan {text, bounds?, sourceId?, lineBoxes?}` (bounds/sourceId/lineBoxes carried for the read-along highlight + tap-to-jump, mirroring the audio-sync glow). A `PageTextSource` produces spans for a page — `TypedTextSource` (pure, reads `page.objects` TextElements) and `PdfPageTextSource` (`lib/services/pdf_text_extractor.dart`, syncfusion `PdfTextExtractor.extractTextLines` → per-line text **with bounds**, extraction offloaded via `Isolate.run` + cached per asset in `PdfTextCache`, like PDF export) ship today. The PDF source **joins a page's lines into one flowing span** (so sentences read naturally across wrapped lines) carrying `lineBoxes` — each line's char-offset-in-span + its page-local rect, scaled from the original PDF page size to the canvas page (imported pages are normalized to the default width). **Adding image-OCR / handwriting-OCR later is one class** implementing `PageTextSource`, registered in `CanvasController._textSources` — nothing downstream changes (the user's explicit future target). Ordering is unified + positional: `orderSpansForReading` sorts every span top-to-bottom / left-to-right within a line (the PDF span's bounds = the union of its lines so it reads ahead of lower annotations; a future OCR source's positioned spans interleave for free), then `readingUnitsForPage` splits each span into per-sentence `ReadingUnit`s (`splitIntoUtterances`) tagged with pageId — and for a span with `lineBoxes` it precomputes each sentence's `rects` (the wrapped-line rects it spans) for the highlight. **Read scope** maps onto the row/page model via `readingOrderPageIds(canvas, mainColumnOnly:)`: default reads **every page row-major** (each row left→right = horizontals included, then next row down); `mainColumnOnly` reads only `row.pageIds.first` of each row (**vertical pages only**). Scope is a device-local pref (`SettingsService.readAloudMainColumnOnly`) toggled by the reader bar's tiny scope selector, which restarts reading with the new scope.

**Engine** (`lib/services/tts_service.dart`): `TtsService` drives one *sentence* utterance at a time and advances on the backend's completion signal. Two load-bearing robustness rules: (1) the next sentence is spoken on a **fresh event-loop turn** (`Future(() => _speakCurrent())`), **never synchronously inside the engine's completion callback** — re-entering the synthesizer from its own callback **crashes the Windows plugin** (Android tolerates it); (2) a `_gen`/`_speakingGen` pair ignores completions from a superseded/paused/stopped utterance (every stop/skip/jump/pause bumps `_gen`, so a stray completion after a `stop()` can't advance the queue). Deliberately does **not** call `awaitSpeakCompletion(true)` (the dual signalling made Windows flakier). Pause = stop + restart-the-current-sentence (robust cross-platform; sentences are short). **Voice selection:** `availableVoices()`/`setVoice(name,locale)` over `flutter_tts` `getVoices`/`setVoice`, persisted device-local (`SettingsService.ttsVoice{Name,Locale}`) and re-applied on init. **Non-Latin auto-switch (fixed 07/19/26):** pinning a manual voice used to force it for *every* sentence forever — e.g. a manually-picked English voice silently ate Sinhala text, which regressed a previously-working "engine auto-picks per script" experience. `_speakCurrent` now calls `_applyVoiceFor(unit.text)` before every utterance: `detectScriptLanguage` (`lib/utils/readable_text.dart`, pure + unit-tested) does best-effort per-utterance script detection (Sinhala/Tamil/Telugu/Kannada/Malayalam/Bengali/Gujarati/Punjabi/Devanagari/Arabic/Hebrew/Thai/Khmer/Myanmar/Japanese/Korean/Chinese/Cyrillic/Greek by Unicode block, majority ≥60% of non-Latin-classified runes, else null) — a **non-null, non-Latin** detection looks up an installed voice for that language in the cached `voices()` list and switches to it (falls back to whatever's already applied if the device has none installed for that language — deliberately best-effort, since voice sets differ a lot across OSes); **null** (Latin-script/ambiguous, the common case) restores the manually-pinned voice. `_appliedLangCode` skips redundant `setVoice` calls across a run of same-language sentences. The manually-pinned voice is therefore really "the voice for my primary/Latin language", not a hard global override. Backends: `_FlutterTtsBackend` (`flutter_tts`, Android/iOS/macOS/Windows) and `_LinuxTtsBackend` (`flutter_tts` has **no Linux plugin**, so shell out to `spd-say -w` / `espeak`; `isSupported` false + no voices if neither installed → the reader toasts a hint). Speed multiplier maps onto `setSpeechRate` (0.5×mult, clamped). Lazily created per controller (`CanvasController.tts`, disposed with it) like the audio services. Android needs the `TTS_SERVICE` `<queries>` intent (added) for package-visibility on 11+; Windows needs `nuget.exe` on PATH at build time (see KNOWN_ISSUES). **`load` vs `speakAll`:** `load(units, startIndex:)` sets the queue **without speaking** (the reader opens paused-at-position); `speakAll` = load + resume.

**UI:** `read_aloud` is in `kOverflowActionSpecs` (`toolbar_actions.dart`) — lives in the "⋯" menu and is **toolbar-customizable on both layouts** (mobile sheet + desktop popup + `_buildPromotedButton` dispatch, all wired). Tapping it runs `CanvasController.prepareReadAloud` **behind the floating `ProgressOverlay` ring** (PDF text extraction can take a moment): it builds the units, stores them in `_readingUnits`, and if non-empty sets `readAloudActive` (a `ValueNotifier<bool>` gating a floating **`_ReaderBar`** in the shared `_buildCanvasArea` Stack, so it shows in normal + full-screen like the audio bars). **Opening the reader does NOT auto-play** — `prepareReadAloud` calls `tts.load` (not speak), so the bar opens paused; the user presses play. The bar mirrors `_AudioPlayerBar`'s look: title + voice picker + tiny scope selector + close; prev/play-pause/next **by sentence** + a cycling speed chip + an `n / total` counter. The **voice picker** (`_pickVoice`) lists the system's installed voices in a searchable sheet **grouped by language** (`_languageLabel` maps a voice's locale like `en-US` → "English (US)" via `_kLanguageNames`; a search box filters by voice name / locale / language) — Android exposes dozens of voices, so grouping + search is how you find one. Empty → a "no selectable voices" snackbar (e.g. Linux). **Resume position:** `prepareReadAloud` opens at the saved position (`SettingsService.readingPositionFor(canvasId)` → matched to a unit by page+source+sentence-start, `_resumeIndex`); the position is saved on close/dispose (`_saveReadingPosition`), so reopening continues where you left off instead of restarting.

**Read-along highlight + tap-to-jump + page-follow (07/19/26):** each `ReadingUnit` carries `charStart`/`charEnd` (the sentence's char range within its source span — for a typed span that's the element's own text), `pageId`/`sourceId`, and precomputed `rects` (for PDF, the wrapped-line rects the sentence spans). `CanvasController.readAloudHighlightNotifier` (`ValueNotifier<({String pageId, List<Rect> rects})?>`, updated from `tts.index`/`speaking` listeners attached when `tts` is first created) holds the current sentence's page-local rects: **typed text** computes them from its element via `selectionRectsForElement` (`text_measure.dart`, uses the **same `TextPainter` layout the painter draws with**, so boxes line up exactly); **PDF text** uses the unit's precomputed `rects`. The painter **merges it into its repaint listenable** and draws a soft accent marker behind the sentence inside each page's translate (like the audio glow — outside the picture cache, a no-op when idle). **Tap-to-jump:** `canvas_screen._maybeJumpRead` — while `readAloudActive`, a tap (non-inking tap paths: the lasso-tap branch + the general `!moved` branch; inking-tool taps still draw) on a **typed** box resolves the char offset via `caretOffsetAt` → `CanvasController.jumpReadingToText`, and on **PDF** text hit-tests the page-local point against the units' line rects via `jumpReadingAtPoint` → `tts.jumpTo(index)`. **Page-follow:** `_followReading` calls `ensureCanvasRectVisible` (pans, **keeps zoom**, moves only when the sentence is off-screen — so it never jitters a visible page; deliberately gentle to stay non-distracting). **Not yet built (tracked in KNOWN_ISSUES "planned", task chips spawned):** general text **selection** (lasso-along-text / drag-select) to **copy + search online**, and long-press-word → **dictionary/translation** in reader mode — both build on this positional layer (the PDF `lineBoxes` + syncfusion word rects give the needed geometry).

**Device status:** read-aloud + highlight + tap-to-jump **confirmed working on Android**. Windows initially crashed (the completion-callback re-entrancy above — now fixed; re-verify on device). macOS/iOS/Linux still unverified. See KNOWN_ISSUES for the Windows `nuget` build prereq + Linux-engine / scanned-PDF-has-no-text-layer caveats.

**Floating canvas overlay bars — structure (load-bearing, fixed 07/19/26):** the recording bar, audio player (`_AudioPlayerBar`), and read-aloud bar (`_ReaderBar`) are **siblings ABOVE the canvas `Listener`/`RawGestureDetector` in `_buildCanvasArea`'s Stack — NOT descendants of it** — and each is wrapped in **`_absorbTaps`** (an opaque `Listener`). Both parts are required: living inside the canvas gesture subtree meant a tap on a bar also went to the canvas's `ScaleGestureRecognizer`, which won the gesture arena and **stole the button's tap** (Android symptom: pause didn't pause; play played for a second then re-paused). Being a sibling above removes the bar from that arena; `_absorbTaps` catches the gaps between buttons so those don't fall through to the page either. Any new floating overlay on the canvas must follow this — put it above the canvas input layer and wrap it in `_absorbTaps` (same idea as the tool-options panel's opaque `Listener`). The `_textEdit` overlay is the deliberate exception: it stays *inside* the canvas `Listener` because its tap handling depends on the canvas pointer handlers (`_pointerInTextEditor`).

### Connections (internal two-way links) (07/19/26, Phase 1)

Link anything to anything inside the store, OneNote-style, with two-way visibility. **Model (`lib/models/link.dart`):** `LinkEndpoint` = an id path (notebookId [+folderId] [+sectionId [+folderId]] [+canvasId [+pageId [+elementIds] | +bookmarkId]]), serialized as a canonical **`omninote://link/n/<nb>[/f/…][/s/…][/c/…][/p/…][/e/id1,id2][/b/…]`** URI — the same string "Copy link" puts on the clipboard, so clipboard/JSON/(future) text-run hyperlinks share one form (`tryParse` never throws on foreign input; distinct from the `omninote://import?id=` share links). `LinkRecord` = one two-way connection {envelope, a, b, label?, aName/bName snapshots, createdAt} — **one record backs both directions**, so removing it from either side removes both. **Registry:** store-root **`links.json`** (id→record map beside notebooks.json), synced + merged by the SAME union-by-id + per-record-LWW + tombstone-delete route (`MergeEngine.reconcile` routes it to `mergeNotebooksIndex`; `isSyncedRelPath`/`listSyncedRelPaths` include it; push uploads it **whole to every connected account** — both `_flushPush` and the resync push-back loop special-case it like notebooks.json). Deletion is a tombstone + rev bump (never a map removal — the same durability rule as the bin); re-linking a tombstoned pair **revives the same record** above the tombstone. `LinkService` (`lib/services/link_service.dart`) is the lazy in-memory view: loads on first query, **reloads only when `SyncService.dataVersion` advanced** (so normal drawing/typing/nav never touches it), queries `linksOf(leafId)` / `linksTouchingCanvas(id)`, mutations persist via `NotebookService.saveLinksJson` (atomic write → journal → upload). **Resolution (`link_resolver.dart`):** `resolveEndpoint` reads only the target's structural files → `ResolvedLink {alive, title, path, kind, reveal}`; soft-deleted/purged/absent targets all resolve **dead** (shown greyed + strikethrough, "restore to re-enable" — aliveness is re-checked at display time, so restore automatically revives links, and no bookkeeping happens on delete). Page aliveness = row membership (no page-file read); page titles are "Page N" by reading order; name snapshots (aName/bName, taken at creation) are the dead-display fallback; live items always show their **live** name + breadcrumb (labels never go stale on rename). **Navigation (`link_navigator.dart`):** resolution produces a `SearchResult`, and `LinkNavigator` is a singleton where the active shell registers its existing search-reveal handler (mobile `_revealFromLink` **pops the root navigator to the shell first** — a link tapped inside a full-bleed canvas must not reveal underneath it; desktop `_revealFromLink` forces `_MainMode.canvas` + expands the sidebar) — so link navigation is byte-for-byte the search-reveal path, glow included. **UI (`widgets/connections_sheet.dart`):** `showConnectionsSheet` = the per-item Connections list (both directions in one list, icon/kind + title + live path, dead rows unclickable, ✕ removes both sides) with header actions **Copy link** + **+ (paste a copied link)** — creation is copy→paste (dedup: same unordered pair = one record; pasting A's link into B and C yields A↔B and A↔C, both visible from A). Entry points: notebook menus (home `_RowMenu` + desktop sidebar popup), `ItemTreeView`'s new optional `onConnectionsLeaf`/`onConnectionsFolder` (wired at all four hosts: sections+folders, canvases+folders, both shells), the pages navigator's per-page menu (Copy link + Connections), and the canvas ⋯ **"All connections"** (`kOverflowActionSpecs` id `connections`, toolbar-customizable, all three dispatch sites) — an **aggregate** view (`aggregateCanvasId`) of every record touching anything in the canvas. **Load-bearing guard:** a Connections sheet opened from *inside* a canvas passes `insideCanvasId` + `onJumpInSameCanvas`; a tapped row targeting that same canvas **jumps in place** instead of revealing — a shell reveal would open a second `CanvasScreen` on the same canvas id (two controllers, one autosave — the split-view rule). Deliberate scope choices: no Connections entry for "this whole canvas" inside the canvas itself (link a canvas from its list row; per user); word-level text links are Phase 3 (`[[` trigger + pencil-edit). Covered by `test/link_model_test.dart` (URI codec round-trips/rejects, record json, links.json union/LWW/tombstone/revival merges). **Account-move re-key follows links (07/19/26):** `rekeyNotebookForMove` collects the old→new id map from `_deepReidNotebookCopy` (`collectIdMap` — notebook/section/canvas levels only; folder/page/bookmark/element ids are preserved by the clone) and `NotebookService.remapLinkIds` rewrites both endpoints of every affected record (rev bump so the rewrite wins LWW; raw-json based so LinkService keeps its one-way dependency; `notifyDataChanged` flushes the cache).

**Phase 2 (07/19/26): element endpoints, landing flash, link items.** **Lasso** selections get **Copy link + Connections** in `buildLassoActionRow` (`tool_option_rows.dart` — covers both the options row and the floating lasso menu): the endpoint is `_selectionEndpoint` (the selected element ids on their page). Element-endpoint queries match by **id overlap**, not leaf equality (`LinkService.linksOfElements`; the sheet routes element endpoints there and resolves "the other side" by intersection) — so re-lassoing a subset/superset of a linked selection still finds its connections. **Landing flash:** element navigation hands ids over via `LinkNavigator.pendingElementFocus` (a one-shot field consumed by `takeFocusFor(canvasId)` — set by the Connections sheet's `_openRow` and by `_openInternalLink`; deliberately NOT part of `SearchResult`). Consumers: `CanvasScreen.initState`'s post-frame jump (cross-canvas arrivals) and `_jumpInThisCanvas` (the same-canvas guard path). `CanvasController.focusElements(pageId, ids)` scrolls the elements' union bounds into view (`ensureCanvasRectVisible`, keep zoom) and sets **`linkFlashNotifier`** (auto-clears after 2s), which the painter merges into its repaint listenable and draws as an accent halo (strokes reuse `_paintAudioGlow`; objects get a blurred rounded-rect outline) — outside the picture cache, no-op when idle; ids that no longer exist fall back to a plain page jump. **Internal link taps:** `canvas_screen._openUrl` routes `omninote://link/` URIs to `_openInternalLink` — same-canvas targets jump/focus in place (split-view rule), others resolve → `LinkNavigator().reveal`. **Link items:** pasting a copied `omninote://link/` URI (the plain-text paste path `_pasteSystemText` → `_pasteLinkItem`) lands `CanvasController.insertLinkItem` — one auto-sized TextElement whose single run carries the URI as `TextRun.link` (display text = the target's live title, trailing non-link space keeps the box grabbable, one undoable op) — and registers the connection (from = the new element, so the target's list shows the placement). **Editing preserves links (load-bearing):** `CharAttr` now carries `link` end-to-end (`attrsFromElement` → edit → `runsFromController`), the editor renders link runs with the painter's exact visuals (`kLinkColor` + underline — decoration doesn't affect layout, so editor/painter parity holds), `linkifyRuns` **passes `omninote://` runs through untouched** (its URL re-scan would otherwise CLEAR them — their display text isn't URL-shaped), and the caret-move style adoption clears `link` from the typing style so typed characters never silently extend a hyperlink. Covered by `test/link_phase2_test.dart` (linkify preservation + idempotence, attrs↔runs link round-trip, insertLinkItem undo/redo + on-page geometry, focusElements flash/fallback). Known gap: undoing a pasted link item leaves its (page-level-alive) registry record.

**Phase 3 (07/19/26): pencil edit, `[[` trigger, in-text links.** **✎ affordance:** `linkPencilSpots` (`text_measure.dart`, pure — same layout the painter uses) yields one small anchor rect per link run, just past its last laid-out box; `CanvasPainter._paintText` draws a ✎ glyph there (inside the picture cache — zero per-frame cost; this **replaced** the old faint-underline "editable space" hint, the `_` the user disliked; the trailing space itself is still appended and still grabbable), and `canvas_screen._linkPencilAt` hit-tests the spots (checked **before** `_urlAt` in both tap paths — pencil and link sit side by side) → `_editLinkAt` opens **`showEditLinkSheet`** (`widgets/edit_link_sheet.dart`: display text + destination fields, Remove link, keyboard-aware viewInsets padding) → `CanvasController.editLinkRun` (one undoable `_stamp`ed op, re-auto-sizes the box) + registry upkeep (`LinkService.removeLinkBetween` old target / `addLink` new target, so retargeting keeps both Connections lists truthful). **`[[` trigger:** `_maybeTriggerLinkPicker` (called from `_onEditingChanged`) detects a just-typed `[[` at a collapsed caret (skips mid-IME composition, `_linkPickerBusy` re-entry guard), then — **post-frame, and it COMMITS the session first** (the picker is a modal sheet whose taps would land outside the TextField and kill the session mid-flow anyway; same re-entrancy rule as `_handleTypingOverflow`) — opens **`showLinkTargetPicker`** (`widgets/link_target_picker.dart`, a search-field + list sheet over `SearchService.buildIndex/filter`) and splices the pick into the committed element **by id** via `CanvasController.insertLinkIntoText` (verifies the `[[` marker is still at the caret, styles the link from the marker's surrounding run via the pure `replaceRunRange` splicer in `text_measure.dart`, appends a plain space, one undoable op) + registers the record. `endpointOfSearchResult` (`link_resolver.dart`) maps a picked SearchResult → endpoint (bookmarks map to their **page** — SearchResult carries pageId, not bookmark id). **In-editor paste** of a copied link now lands as a live hyperlink too: `_handleEditorPaste` routes an `omninote://link/` text through `RichTextController.insertLinkText` (inserts the target's name, stamps `link` onto the inserted attrs, plain trailing space) + registers the record — closing the Phase 2 gap. **Bookmarks are directly linkable:** each row in the Bookmarks sheet has a copy-link button (endpoint carries `bookmarkId`, resolving to the bookmark's name + page). **Connections "+" is now a two-item menu:** Paste copied link / **Choose target…** (the same picker `[[` uses — free once it existed). Covered by `test/link_phase3_test.dart` (replaceRunRange splicing, insertLinkIntoText marker-guard + undo, editLinkRun text/destination/remove + undo, linkPencilSpots geometry, insertLinkText attrs, external-endpoint codec).

**Design round (07/20/26, user feedback after device testing):** **(1) Mobile link-nav stops one level up** — `MobileShellScreen._revealFromLink`: a link to a *container* (notebook/folder/section/canvas — discriminated by `r.pageId == null`) lands on its **parent list with the target glowing** instead of auto-opening it (opening lost the "this is what was linked" context on mobile); notebook targets glow the home card via **`HomeScreen.glowRequest`** (a static one-shot `ValueNotifier` the kept-alive tab root listens to). In-canvas targets (page/element/bookmark — pageId set) still open the canvas, where the landing flash IS the indication. Desktop keeps full reveal. **(2) Linking a lasso selection drops a visible marker** — the selection's Connections sheet passes `onAddTarget` (a `showConnectionsSheet` hook): adding a target inserts a link item (`insertLinkItem` gained `nearBounds` — anchored just under the selection, page-clamped, movable like any text element) and registers the record with **the marker's id included in the element side** — which is what lets the marker's ✎ retarget that very record. Dedup for these is by element-id overlap (the exact-pair check can't see the marker id). **(3) One unified link system** — `LinkTargetKind.external` + `LinkEndpoint.external(url)` (empty notebookId, `toUri()` = the raw URL, so records/clipboard/JSON need no schema change; `LinkEndpoint.sideFrom` auto-detects internal URI vs URL vs garbage — used by record JSON *and* the sheet's paste, per the original "don't ask the type" spec). External rows show a globe + open the browser directly (sheet stays up); they're always "alive". **Every Connections row now has a ✎** → the same `showEditLinkSheet` the canvas pencil uses: rename (a custom name persists as `record.label`; retyping the resolved title clears it) and/or **retarget** — internal↔external freely swapped via `LinkService.updateLink` (sides are now mutable; rewritten in place + rev bump, so the record id and pairing survive). The canvas ✎ handler routes through **`retargetByElement`/`removeByElementTo`** (matches records whose element side *contains* the tapped element — needed because a marker's record also carries the ink ids), falling back to `addLink` for runs with no record yet (e.g. upgrading a plain auto-linkified URL into a tracked connection).

**Design round 2 (07/20/26): scroll-to-glow + reciprocal markers.** **(1) A glowing list row scrolls itself into view** — the highlight was pointless off-screen in a long list. `ScrollIntoViewOnce` (`lib/widgets/scroll_into_view.dart`) fires one post-frame `Scrollable.ensureVisible` (alignment 0.35, walks all ancestor scrollables) when first built; it wraps every glow surface: `ItemTreeView`'s leaf **and** folder rows (covers sections/canvases on both shells — the row builders were split into `_buildLeafRow`/`_buildFolderRow` so the glow wrap stays a one-liner), the desktop sidebar's notebook-header glow box, and the mobile home card's glow overlay — where `_onGlowRequest` additionally does a **coarse `jumpTo` first** (index × estimated row height) because `ReorderableListView.builder` is lazy and an unbuilt off-screen card has no context to `ensureVisible` (the trees are shrink-wrapping Columns, always built once expanded, so they don't need this). Key it by the glow target so a re-reveal re-triggers. **(2) Linking shows on BOTH canvases** — creating a connection whose **target** is an element endpoint (pasting a copied lasso/element link) drops a **reciprocal marker** next to the target elements too, a link item pointing back at the source side, so item A shows the link when you paste A's link near item B. `LinkService.addLinkWithReciprocalMarker` (dedup by element-id overlap first, then `dropMarkerNear` + the marker id folded into the record's target side — so that marker's ✎ retargets the record) wraps `addLink` at every flow where the target can be an element endpoint: canvas paste (`_pasteLinkItem`), in-editor link paste (`_handleEditorPaste`), the ✎ fresh-record fallback (`_editLinkAt`), the Connections sheet's default `_addTarget`, and the lasso `onAddTarget` (the `[[` picker stays plain `addLink` — search results are never element endpoints). Marker placement routes **open-canvas-first**: `SyncService.insertMarkerInOpenCanvas` → `CanvasSyncListener.onInsertMarker` → the live controller's `insertLinkItem` (in memory + undoable; same-canvas paste — the common page-1↔page-3 case — lands here), else `NotebookService.addLinkMarkerToPage` edits the page file directly through `savePage` (rev bump → syncs; a disk write under an open canvas would be clobbered by its autosave — the same dispatch rule as page restore). **Markers never overlap:** several connections onto the same item anchor at the same spot, so both placement paths run the candidate rect through `stackBelowObstacles` (`text_measure.dart`, pure + unit-tested in `link_phase2_test`) — nudged straight down past any existing text box (each round jumps below the LOWEST intersecting one), so repeated markers stack into a vertical list; page-bottom clamp last (an extreme-bottom overlap beats an invisible marker). Known gap (same family as the pasted-link-item one): undoing the source-side paste doesn't remove the record or the reciprocal marker on the other canvas (cross-canvas undo can't reach it).

### Split view (in-app multi-canvas) (07/18/26)

Work on 2+ canvases at once in one window. The reusable **`CanvasWorkspace`** widget (`lib/screens/canvas_workspace_screen.dart`) hosts the panes; it's used by both the **mobile route** `CanvasWorkspaceScreen` (routed from `section_screen`/`mobile_shell`/`note_search` instead of pushing `CanvasScreen`) and the **desktop shell's main pane** (`soloNonEmbedded: false` — so split works on desktop/tablet too; selecting a different canvas in the list resets the split via the id-keyed workspace). **Load-bearing design:** with a *single* pane and `soloNonEmbedded` (mobile) it renders the ordinary **non-embedded** `CanvasScreen` — identical to before (bottom-sheet menus, auto back button) — so the 99% case can't regress; it only gains an "Open canvas alongside" action. Adding a pane asks a **direction** first (`pickSplitDirection` → `SplitDir.sideBySide` = Row/vertical divider, or `SplitDir.stacked` = Column/horizontal divider; a differing pick flips the whole axis), then a **canvas picker** (`canvas_picker.dart`) that **opens directly in the source pane's section** (`startNotebookId`/`startSectionId` deep-link, Back walks up to sections → notebooks) and greys out already-open canvases. Multiple panes render as a Row/Column of **embedded** `CanvasScreen`s with **draggable dividers** (`onPanUpdate` → per-pane `flex`, 220px min, row/column resize cursors). `CanvasScreen` gained three **optional, null-by-default** hooks the workspace supplies: `onSplitRequested`, `onClosePane` (secondary pane leading = close ✕), `onBack` (primary pane leading = back, only on the mobile route). The `_paneLeading()` helper + `automaticallyImplyLeading: mobile && paneLeading == null` pick the leading. The split action lives in `kOverflowActionSpecs` (**toolbar-customizable**; gated by `shown('split') && onSplitRequested != null`, empty outside a workspace via `_buildPromotedButton`). **Never open the same canvas in two panes** — two `CanvasController`s on one canvas id fight over the debounced autosave and the id-keyed `CanvasSyncListener` registry (the picker + `_addPane` both guard against it). `PopScope` closes the last pane before leaving. **Not yet built:** true OS multi-window (separate app windows — see KNOWN_ISSUES; deferred, needs multi-engine sync coordination).

### Android multi-window (separate app windows) (07/18/26, foundation — not device-verified)

Open **separate OminiNote windows** (own Recents entry / side-by-side on large screens/DeX). Native: `android:resizeableActivity="true"` + `MainActivity`'s `omninote/multiwindow` channel (`isSupported` = API 24+, `openNewWindow` launches another `MainActivity` with `NEW_TASK|MULTIPLE_TASK|NEW_DOCUMENT`). Dart: `MultiWindowService` + a Settings → Multi-window → "Open new window" tile (Android N+ only). **The load-bearing safety** (`lib/services/sync_coordinator.dart`): each window is its own `FlutterEngine`/isolate with its **own** `SyncService`/`NotebookService`/`SettingsService` singletons over one filesystem, so two windows both syncing = two writers on `drive_index.json`/`sync_journal.json` = corruption. `SyncCoordinator` elects **one sync owner** via a heartbeat lease file (`sync_owner.lock`, appDir root, not a synced path): the owner runs GC + the Drive loop; non-owners skip it. `SyncService._active` (set by `init()`, which only the owner calls; `init` is idempotent) gates `_markDirty` — a non-owner **never journals or pushes**. A lone window wins the lease instantly → **single-instance behavior is byte-for-byte unchanged** (the critical invariant). `main.dart` runs GC + `SyncService().init()` only for the owner (or `onBecomeOwner` takeover), gates the resume/background hooks on `active`, and releases the lease on `detached`. `leaseHeldByOther` (take-over decision) is pure + unit-tested (`test/sync_coordinator_test.dart`). **Known limits (KNOWN_ISSUES):** a non-owner's edits stay local until it becomes owner + the file is re-touched (needs per-instance journals — deferred, requires device testing); same canvas in two windows is LWW; **the whole feature is un-verified on real Android multi-window** — build fail-safe but test on a device before relying on it.

### Gotchas

- Elements/Section/pages are mutable by design; **undo correctness depends on ops snapshotting `deepCopy()`s** — never store live element refs in an op.
- `CanvasController.setScreenSize` is called during build → it must never `notifyListeners()` synchronously.
- Stroke colors are absolute; canvas page backgrounds are user-chosen (default white), so theme changes don't affect page/ink contrast anymore.
- On macOS, `com.apple.security.app-sandbox` is enabled but no file-access entitlement is granted yet — file picking may fail in release builds (see KNOWN_ISSUES).
- **Never mutate `sf.PdfPageSettings.size` (or `.orientation`) on an existing settings object in `pdf_exporter.dart`.** The setter re-derives width/height as `(min, max)` of the *current* orientation (default `portrait`), silently swapping them whenever width > height — which is every horizontally-merged row and every landscape-shaped PDF page. Always assign a **fresh** `sf.PdfPageSettings(Size(w, h))` (size-only constructor arg — no orientation) to `section.pageSettings` instead; that stores the size verbatim. Covered by `test/pdf_export_test.dart` — extend those tests if the exporter's page-sizing code changes.
