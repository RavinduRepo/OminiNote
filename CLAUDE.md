# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Omininote (`omininote`) is a Flutter stylus note-taking app. Structure: **Notebook → Section → Canvas**, where a Section opens into a freely pan/zoomable **Canvas** of many **pages** (blank or PDF-backed) that the user inks, types, and pastes images onto — Samsung-Notes-style, with vector PDF export. Target platforms are Android, Windows, macOS, and Linux (iOS is not a priority; web is not configured). The full feature spec is `CANVAS_SPEC.md`; the user's raw intent notes are `PAGE_BEHAVIOR_REQ.md`; open bugs live in `KNOWN_ISSUES.md` — keep all of these in sync with code changes.

## Commands

```bash
flutter pub get                 # install dependencies (run after editing pubspec.yaml)
flutter analyze                 # static analysis / lints
flutter test                    # run all tests
flutter test test/models_test.dart    # run a single test file
flutter devices                 # list run targets (physical devices show here even if adb isn't on PATH)
flutter run -d <device-id>      # run on a device from `flutter devices`
flutter run -d windows          # run desktop build directly
flutter build apk / windows / linux / macos
```

Sync builds need the gitignored `.dart_defines.json` (OAuth client ids; template in `.dart_defines.example.json`, real values in `SECRETS/`): `flutter run -d <id> --dart-define-from-file=.dart_defines.json`. The `Makefile` wraps this per platform (`make android|linux|windows|macos`); on Windows invoke it as `mingw32-make` from Git Bash.

No custom lint config beyond default `package:flutter_lints/flutter.yaml`.

**OneNote importer** (`tools/onenote_importer/`, own README): recovers `.onepkg`/`.one` notes into the app's on-disk store — a Rust extractor built on Joplin's OneNote parser (source npm-fetched into gitignored `.cache/`) dumps `extract.json` + lossless assets, `preview.html` visualizes it, and `convert.dart` (pure Dart, run via `dart run`) emits/installs the store. Section groups → notebook `FolderNode`s, sub-pages → canvas-tree `FolderNode`s, and each infinite OneNote page is tiled into rows/pages with cuts only where no element crosses. Covered by `test/onenote_import_test.dart` (tiling invariants + model round-trip of converted output).

## Architecture

Single-package Flutter app, no state-management library (plain `ChangeNotifier` + `setState`), no backend — persistence is local JSON files. Key dependencies: `pdfrx` (PDF page **rendering only** — its viewer/gestures are not used), `perfect_freehand` (ink outlines), `syncfusion_flutter_pdf` (vector PDF export; Community License), `file_picker`, `path_provider`, `crypto` (content-addressed assets), `super_clipboard` (OS image/HTML clipboard), `html` (parsing pasted rich-text HTML into styled runs).

**Navigation (mobile — bottom-nav shell):** `MobileShellScreen` (`lib/screens/mobile_shell_screen.dart`) is the mobile root (picked by `_RootRouter`). It's a **bottom navigation bar** over four destinations — **Notebooks / Search / Bin / Settings** — where each tab owns its **own nested `Navigator`** (a `GlobalKey<NavigatorState>` per tab, kept alive via a `PageView` + `AutomaticKeepAliveClientMixin` wrapper). **Tabs slide horizontally**: tapping a tab `animateToPage`s toward it, and a horizontal **swipe** moves between tabs — but only at a tab's root (a per-tab `NavigatorObserver` recomputes `_swipeEnabled` post-frame — never synchronously, or the initial-route push at startup would `setState` during build — and locks `PageView` physics to `NeverScrollableScrollPhysics` once you've drilled in, so drags belong to the content). The notebook drill-down (`HomeScreen` notebooks → `NotebookScreen` sections → `SectionScreen` canvases) pushes **inside the Notebooks tab's** nested navigator via **`slideRoute`** (horizontal slide-in from the right + parallax-dim of the page below — `app_theme.dart`, distinct from the `fadeThroughRoute` used elsewhere), so the bottom bar **stays visible the whole way down**. The **Bin tab reloads on entry** (`BinScreen.refreshSignal` — a `ValueNotifier` the shell bumps whenever the Bin tab is opened; the kept-alive tab's `initState` load otherwise never re-ran, so items deleted elsewhere wouldn't appear). **Perf (07/13/26):** that entry-reload used to rescan the whole store (`listDeletedItems` walks every notebook/section/canvas + a head-read of every page file) *every* time — the main cause of swipe-into-Bin jank. Now (a) the shell **debounces the `refreshSignal` bump ~350ms** (a `Timer` in `_onEnterTab`) so the scan can't compete with the slide animation, and (b) `BinScreen` **caches its list and only rescans when the store changed** — it captures `SyncService.dataVersion` before each scan and `_maybeReload` skips the scan unless `dataVersion` advanced. For this to stay correct, **local** delete/restore/purge now bump `dataVersion` too (previously only remote pulls did): `notifyDataChanged()` is called from `deleteNotebook`/`deleteSection`/`deleteCanvas`/`deleteSectionFolder`/`deleteCanvasFolder`/`restoreBinItem`/`purgeBinItem`, and from `savePage` **only when the page carries a tombstone** (`deletedAt != null` — catches `CanvasController.deletePage` without firing on ordinary drawing autosaves or the GC's `_writeAtomic`-direct purge). The first load (and the Settings→Deleted-items path, which has no `refreshSignal`) runs via a post-frame callback so it's off the push-animation frame. Opening a canvas pushes `CanvasScreen` on the **root** navigator (`Navigator.of(context, rootNavigator: true)`), which covers the shell → **the editor is full-bleed with no bottom bar** (the deliberate exception; everything else keeps the bar). The bar is a custom `_MobileTabBar` (neutral surface + hairline top border, icon-over-label going amber when active — no Material pill). `PopScope` routes the system back button: pop the active tab's nested navigator first, else jump to the Notebooks tab. **Search is now a tab** (`NoteSearchView`, the embeddable form of the search screen — rebuilds its index on `SyncService.dataVersion`, **debounced 400ms** so a burst of changes coalesces into one full-store reindex; **07/13/26:** local create/rename (`createNotebook/Section/Canvas`, `createSectionFolder/CanvasFolder`, `renameNotebook/Section/Canvas`) now bump `dataVersion` too — previously only remote pulls + deletes did — so a freshly-**added** item is searchable without an app restart; the field focuses only on tab entry (`focusSignal`) and is **unfocused when you leave the Search tab (`onPageChanged` to a non-Search tab) or open a result (`_open`)**, so the keyboard never re-pops when returning to a list from a canvas; an **empty query renders a hint, not the whole index** — `_results` is `[]` until you type; the per-keystroke `filter` is **debounced 140ms for large indexes** (`≥400` results) and instant below that, with a flush-on-submit so Enter opens a current result); `SearchService.buildIndex` reads its section/canvas files **bounded-concurrently** (`_mapBounded`, 8 in flight) instead of sequentially — the small structure-only decode stays on-main (offloading wouldn't pay: the models must be built here to navigate to results); picking a result calls the shell's `_revealSearchResult`, which switches to the Notebooks tab, rebuilds its drill-in stack (glow trail preserved via `glowId`), and opens any canvas on the root navigator — mobile now uses the same `onReveal` hook the desktop shell always did (no more root-navigator `popUntil` rebuild). **Bin is a tab** too (the Settings → Deleted items row still exists as a secondary path). Search/Settings are therefore **gone from the list screens' app bars** (`HomeScreen`/`NotebookScreen`/`SectionScreen` keep only the `+`). All list screens: **no FABs**; adding goes through the app-bar `+`. Each list supports **pull-to-refresh** (`RefreshIndicator` → `SyncService().syncNow()`; `RefreshableEmpty` keeps the gesture on empty states). The canvas itself has no pull-to-refresh. **Note:** `NotebookScreen`/`SectionScreen` still `searchRouteObserver.subscribe` for glow-on-pop-back, but that observer is on the *root* navigator while they now live in a nested one, so the re-glow-on-return trail no longer fires (the initial `glowId` glow still does) — an accepted minor regression of the nested-navigator shell. **Mobile kebab menus are bottom sheets:** `showActionSheet` (`lib/widgets/action_sheet.dart`) renders a floating rounded sheet (grab handle, icon+label rows, destructive in the error color); `ItemTreeView`'s leaf/folder kebabs use it when `!dense` (mobile) and keep `PopupMenuButton` when `dense` (desktop), home's notebook `_RowMenu` always uses it (mobile-only), and the canvas overflow menu uses it when `!widget.embedded`. **Bin + search rows** carry the same colored identity rail as the tree (bin: `AppPalette.resolveColor` of the item's most-specific id; search: canvas→section→notebook).

### Responsive shell (`main.dart`'s `_RootRouter`, `lib/screens/desktop_shell_screen.dart`)

`MaterialApp.home` is `_RootRouter`, which picks between the mobile push-navigation flow above and `DesktopShellScreen` — a OneNote-desktop-style view with a **60px left icon nav rail** (`_buildNavRail`: logo, then Notebooks / Search / Bin destinations, with Settings + an account avatar pinned to the bottom — the v2 redesign's rail; `palette.canvas` background) followed by **three panes**. The rail is wired to existing behavior (per the user's scope choice, not a pane-switcher rework): **Notebooks** toggles the left panes collapsed/expanded (mirrors the header book icon), **Search** opens the overlay, **Bin/Settings** push their screens — so Search + Settings were removed from the sidebar header, which keeps only the collapse toggle + `+` + sync. The build's pane-width math subtracts the 60px rail from `constraints.maxWidth`. The three panes: a resizable left sidebar showing the notebook → super-section → section tree (see "Item tree" below), a **resizable canvas-list column** (`_buildCanvasColumn`, OneNote's page list; default 230px, drag-resizable 170–380px via a second `_resizeDivider`; `palette.surface2` background, dense `ItemTreeView<Canvas>`, slides closed via `AnimatedContainer` when nothing is selected — the divider only renders while open), and `CanvasScreen` embedded directly in the main pane (no pushed route, no breadcrumb — the list stays visible beside it; panes are keyed by canvas id so state resets cleanly). Notebooks load into state as mutable `Notebook` objects (the in-memory source of truth: tree drags/collapses mutate them in place then `saveNotebook`); operations that touch section *files* call `_reloadNotebook(id)` to resync. The header's book icon (`_buildSidebarToggle`) collapses **both** left panes (`_sidebarCollapsed` shrinks the sidebar to a narrow rail and slides the canvas column to width 0) showing just a colored initial per notebook — tapping one expands back and jumps straight to it; the resize-drag divider only renders while expanded. **Animation gotcha (fixed):** the sidebar content is laid out at its *target* width inside `ClipRect`+`OverflowBox` so the width animation clips/reveals rather than reflowing the children mid-transition (an unclipped `AnimatedContainer` reflowed the header `Row` at intermediate widths → visible overflow artifacts on expand). A 20px top inset keeps the toggle off the very top-left window corner. The choice is `LayoutMode.auto` (window width ≥ 840 logical px via `LayoutBuilder`, live on resize), or a manual `mobile`/`desktop` override — both persisted via `SettingsService.layoutMode` and editable from `SettingsScreen`'s "Layout" section. `CanvasScreen` takes an optional `onSectionRenamed` callback so the sidebar can invalidate its cached section list when a rename happens from *inside* the embedded canvas (the sidebar's `NotebookService.getSections()` call returns fresh `Section` objects on each fetch, decoupled from the mutable instance `CanvasScreen` edits in place — without the callback the sidebar would show the stale name). **Gotchas:** never write `setState(() => someMap.remove(key))` as a single-expression arrow body when the map's value type is a `Future` — the arrow's implicit return value becomes that `Future`, and `State.setState` asserts against callbacks that return one (`"setState() callback argument returned a Future"`). Use a block body (`setState(() { someMap.remove(key); })`) instead. Also: the sidebar's `AnimatedContainer` width is wrapped in a `LayoutBuilder` that clamps it to `constraints.maxWidth` — don't remove that clamp, since some Android devices briefly report a near-zero-width frame at startup, and an unclamped fixed-width sidebar overflows its `Row` for that frame (found on a real tablet, not just theoretical).

### Theming (`lib/theme/app_theme.dart`)

All colors/shapes flow through `AppTheme.light()/dark()` + the `AppPalette` `ThemeExtension` (read via `Theme.of(context).extension<AppPalette>()!`). Amber accent, rounder cards (`kRadius`=10 — was 6; the "v2 redesign" softened the old crisp Graphite shapes). The **dark** theme was retuned to the v2 mockup's deep charcoal (scaffold `#12151B`, cards `#171B22`, `--line` borders `#2A313D`, ink `#E9ECF1`, amber `#E9A23B`, drawing ground `#14171D`); the **light** theme still uses the original Slate palette (the mockup is dark-only). Never pure black. `SettingsService` (`settings.json`) exposes a `ValueNotifier<ThemeMode>` (live System/Light/Dark) **and** a `ValueNotifier<PageBackground> defaultPageBackground` (app-wide default page color + pattern); both are edited from `SettingsScreen`, and `createSection` seeds a new section's default background from the latter. Dates format MM/DD/YY and counts get thousands separators via `lib/utils/formatting.dart`.

### Item tree (`lib/widgets/item_tree_view.dart`, `color_swatch_picker.dart`)

`ItemTreeView<T>` renders a container's recursive contents (leaf items + nested super-section folders) as a drag-reorderable, collapsible tree; it's generic and reused at two levels — sections inside a notebook (desktop sidebar `dense: true`, mobile `NotebookScreen` `dense: false`) and canvases inside a section (desktop canvas-list column `dense: true` + mobile `SectionScreen`). It flattens the tree (respecting each group's `collapsed`), renders each row as a `LongPressDraggable`, and places `DragTarget` insertion-gaps between rows (reorder within a parent) plus makes each group header a `DragTarget` (drop *into* the group, at its end); a drop whose `containerId` differs routes to `onCrossDrop`. It mutates the passed-in `nodes` **in place** then calls `onTreeChanged` to persist; all other actions (rename/color/delete/add/open/relocate) are delegated to callbacks so the host owns dialogs + service calls. `_wouldCycle` blocks dropping a group into its own subtree. The view returns a shrink-wrapping `Column` (not a scrollable) so it composes inside each expandable notebook within the host's single scroll view. Colors are set via `showColorSwatchPicker` (curated `AppPalette.swatchColors`; returns a `ColorChoice` whose null `color` means "use the default identity color"). **Visual hierarchy** (deliberate, both themes, OneNote-style — *rows stay neutral, color lives only in small elements*; full-row color washes were tried and rejected as "too much color"): folder rows = chevron + filled folder icon in the folder's color + w600 text on a transparent row; leaf rows are lighter — a short colored identity pill (3px, indents with the tree) + dim icon + w400 text; nesting depth is drawn as hairline vertical indent guides (`_indentGuides`) — don't collapse these back into uniform rows, telling the levels apart was a user complaint. Notebook-level reordering is a separate `ReorderableListView` (`buildDefaultDragHandles: false`) with the whole notebook row/card wrapped in `ReorderableDelayedDragStartListener` — long-press to reorder, **no visible drag handle** (the old `drag_indicator` icons looked messy; notebook headers aren't tree rows, so this doesn't fight the tree's `LongPressDraggable`). Desktop sidebar notebook rows carry a 20px solid initial chip (the only colored element; row bg is `surface2` while expanded, else transparent); the expanded section tree below is indented under a 2px rail in the notebook's color. Mobile home cards use a 38px solid initial chip on a neutral card.

### Data model (`lib/models/`)

All geometry is stored in **PDF points** (1pt = 1/72"; default page A4 595×842) so export maps 1:1.

- `Notebook`: id, name, createdAt, optional `color` (ARGB int; null → deterministic `AppPalette.identityColor`), and a **recursive contents tree** `nodes: List<NotebookNode>`. `NotebookNode` (sealed, in `notebook.dart`) = `SectionNode {sectionId}` (leaf) | `GroupNode {id, name, color?, collapsed, children}` — a **super-section** (OneNote "section group") that holds sections *and* nested groups, arbitrarily deep. `allSectionIds` flattens depth-first. Legacy notebooks with a flat `sectionIds` list auto-upgrade to a flat tree of `SectionNode`s on load; `toJson` still writes a derived flat `sectionIds` for any back-compat reader.
- `Section` (`section.dart`): the document; owns `rows: [PageRow]` (each row = horizontal band of `pageIds`), `defaultPageSize/Background`, `attachments`, optional `color` (like Notebook). **Mutable** (controller edits it in place).
- `CanvasPage` (`canvas_page.dart`): one sheet; size, `PageBackground` (color + `BgPattern` blank/ruled/grid/dotted), optional `PdfSource {assetId, pageIndex}` (= PDF-backed page), `elements` list (z-order = list order).
- `CanvasElement` (`element.dart`): **sealed, deliberately mutable** — `StrokeElement` (tool pen/highlighter, color, size, points with pressure `p`, transient `cachedOutline` Path invalidated on mutation), `TextElement` (rect, rotation, `runs: List<TextRun>` for per-range styling — see Text below — plus baseline fontFamily/size/color/bold/italic/align as fallback and toolbar default), `ImageElement` (rect, rotation, assetId), `AttachmentElement` (rect, rotation, assetId, name, mime — a visible "attached file" chip; "Add as attachment" drops one on the current page, tapping it in lasso/text mode opens the file via `open_filex`; export draws the chip, embeds the file as a PDF document attachment, and overlays an invisible `PdfActionAnnotation` + `PdfJavaScriptAction` running `exportDataObject({cName, nLaunch:2})` so clicking the chip in Acrobat/Foxit-class readers opens the embedded file — embed names are de-duplicated per assetId since `exportDataObject` looks up by name; simpler viewers still reach it via the attachments panel). All support `deepCopy({withNewId})`, `translate`, `scaleBy` (uniform; no-op for `TextElement`'s size — see Text), `rotateBy`. Ids from `newModelId()`. All carry a `zIndex` (json `zi`, default 0): strokes and objects live in separate lists (sync model), so painter **and** exporter draw `zOrderedElements(page)` — the combined list stable-sorted by zIndex (ties keep strokes-under-objects) — letting Send-to-back put an image *behind* ink; `_reorderSelection` sets `max+1`/`min-1` and `_stamp`s the elements so the z change syncs. **Because the tie-break renders objects *over* strokes at equal z, a plain-inserted image lands on top of ink** — so freshly added/pasted images go through `CanvasController.addImageBelowInk`, which sets the image's z just below the lowest stroke z (default 0 → image at −1) so ink (now or drawn later) stays on top while the image still sits above the page background; images tie among themselves and fall back to insertion order (newest on top). Text/attachments still insert on top via `addElement`. When adding a subtype: include `zi` in toJson/fromJson/deepCopy or layering silently resets on reload. Adding a subtype breaks sealed-switch exhaustiveness in `canvas_painter`, `canvas_controller` (`_lassoHits`, `applyColorToSelection`), and `pdf_exporter` — the compiler flags every site.

### Persistence (`lib/services/notebook_service.dart`)

```
<appsupport>/notebooks.json                                        # notebookId -> Notebook
<appsupport>/notebooks/<nbId>/sections/<sectionId>/section.json    # structure only (rows, defaults, attachments)
<appsupport>/notebooks/<nbId>/sections/<sectionId>/pages/<pageId>.json   # one CanvasPage per file
<appsupport>/notebooks/<nbId>/sections/<sectionId>/assets/<sha256>.<ext> # content-addressed PDFs/images (dedup)
```

`<appsupport>` is `getApplicationSupportDirectory()` — the OS **application-support** dir, deliberately **not** `getApplicationDocumentsDirectory()`: on desktop the latter maps to the user's real `~/Documents` and dumped the whole store (JSON + `settings.json` + `sync_journal.json` + `drive_index_*.json`) loose in there. App id is `io.github.ravinduRepo.omininote` on every platform (Android `applicationId`/`namespace`, macOS `PRODUCT_BUNDLE_IDENTIFIER` + entitlements keychain group, Linux CMake `APPLICATION_ID`; Windows has no bundle id so `Runner.rc` sets `CompanyName=io.github.ravinduRepo` + `ProductName=omininote`). `<appsupport>` resolves to: Windows `%APPDATA%\Roaming\io.github.ravinduRepo\omininote`; macOS `~/Library/Containers/io.github.ravinduRepo.omininote/Data/Library/Application Support/io.github.ravinduRepo.omininote` (sandboxed); Linux `~/.local/share/io.github.ravinduRepo.omininote`; Android `/data/user/0/io.github.ravinduRepo.omininote/files` (private internal). All three data-store call sites use it — `NotebookService.init`, `SettingsService.init`, `DriveService._loadIndex/_saveIndex`; only `notebook_share_ui.dart`'s export staging stays on `getTemporaryDirectory()`. iOS is still `com.example.omininote` (not a target). No data migration exists — this changed the base dir with no in-app move, so a pre-change install's old `~/Documents` store is orphaned (acceptable: pre-release, no real users).

All writes are **atomic** (`_writeAtomic`: tmp file + delete + rename). Editing a page rewrites only its file; structural changes rewrite `section.json`. **Page-JSON is (de)serialized off the main isolate (07/13/26, perf):** `jsonEncode`/`jsonDecode` of a page's full stroke/object payload is pure CPU that used to jank drawing (the debounced autosave) and canvas-open (decoding every page). `savePage` now offloads the encode via `Isolate.run` **only for dense pages** (`strokes+objects ≥ _kPageOffloadElements`=120) and **never for tombstone/purge saves** (they stay inline so a delete-save can't lose a write-ordering race to an in-flight pre-delete autosave — `CanvasController.deletePage` fires its save via `unawaited`); light pages encode inline (the isolate spawn+copy would cost more than the work). `loadPages` reads page files in bounded chunks (`_kPageReadChunk`=32, so peak memory doesn't hold the whole canvas) and batch-decodes each chunk via `Isolate.run` when it's heavy (`_kPageDecodeChars`=256KB), else inline — then builds the models + prunes tombstones on the main isolate (`fromJson` touches ui/model objects that can't cross the boundary). `toJson` also stays on-main (it walks live elements whose transient `cachedOutline` `Path` isn't sendable); only the resulting pure-data map is shipped. The decode helper `_decodePageJsonsSync` is **static** so its `Isolate.run` closure can't drag `this` (File handles, caches) across the boundary. Both offloads fall back to inline on any isolate failure — a hiccup must never drop a save/load. The notebook's whole contents tree (sections, super-sections, order, colors, collapse state) lives in `notebooks.json` — `saveNotebook(notebook)` persists it; `reorderNotebooks` rewrites the index object in the new key order (JSON object order is preserved through decode/encode). Group ops are tree-walk helpers (`_findGroup`/`_removeSectionNode`/`_spliceOutGroup` for ungroup/`_removeGroupNode`); `deleteGroup` also deletes its contained sections' dirs, `ungroup` splices a group's children into its own position (keeping sections). A legacy pre-canvas `notebooks.json` (detected by `pageIds` keys) is moved to `notebooks_legacy.json` and the app starts fresh (spec §16). Page files orphaned by deleted pages are left on disk (undo may restore them; harmless). `RenderCache` (`render_cache.dart`) renders PDF pages via pdfrx engine (`PdfDocument.openFile` → `page.render` → `createImage`) into an LRU of `ui.Image`s bucketed by zoom, plus decodes raster image assets; `pdfrxFlutterInitialize()` is called in `main()`.

**Note:** the on-disk shape now has a **Canvas** level between Section and pages — `notebooks/<nb>/sections/<sec>/canvases/<cid>/canvas.json` (rows/defaults/attachments) + `.../canvases/<cid>/pages/<pid>.json` + `.../canvases/<cid>/assets/<sha>.<ext>`. Section is a tree of canvases; Canvas owns the rows. All five model types (Notebook/Section/Canvas/CanvasPage/CanvasElement) carry a sync **envelope** (`schemaVersion,id,rev,updatedAt,deviceId,deletedAt`); save methods call `bumpRev(deviceId)` so `rev`/`updatedAt` actually advance (they're the LWW signal). `CanvasPage` uses the **stroke-set model** (`strokes` + `erased` tombstones + `objects`), not a flat `elements` list.

### Cloud sync (`lib/services/sync_service.dart`, `drive_service.dart`, `auth_service.dart`, `desktop_oauth.dart`, `sync/merge_engine.dart`)

Two-way Google Drive sync (spec: `implementation_plan.md`, status: `task.md`). The Drive tree **mirrors** local storage under a visible `omininote/` My-Drive folder (scope `drive.file`). **Auth:** Android via `google_sign_in` (silent restore); desktop via loopback **PKCE** OAuth (`desktop_oauth.dart`) with the `refresh_token` in `flutter_secure_storage` and the ~1h access token refreshed proactively (5 min pre-expiry) + reactively — `_AuthedClient` re-fetches headers per request so the fresh token is always used. **`DriveService`** keeps a persisted `drive_index.json` (relPath ↔ fileId ↔ `headRevisionId`) so updates are O(1) and our own uploads are echo-suppressed; `listAllFiles` does a single paginated `files.list` and rebuilds relative paths from parents (treating every "omininote" folder as a root → duplicate-root create/create races self-heal). **`MergeEngine`** is pure/unit-tested (`test/merge_engine_test.dart`): `notebooks.json` = union map by notebook id + per-id LWW (so two devices' notebooks both survive first sync); `section.json`/`canvas.json` = single-doc LWW by `(rev,updatedAt,deviceId)`; page files = set-union of immutable strokes + grow-only erase tombstones, LWW for text/image objects filtered by a grow-only **object-delete tombstone** set (`CanvasPage.deletedObjects`), plus background. **Deletion is always soft-delete, never a hard remove/splice** — this is the load-bearing rule that makes deletes durable (v3 fix, see KNOWN_ISSUES "Fixed"): `deleteNotebook`/`deleteSection`/`deleteCanvas` set `deletedAt`+bump `rev` and keep the entry/file on disk (a hard-removed entry has no envelope left to win a merge comparison, so a live remote copy resurrects it — this was the actual bug, not a design gap); `getNotebooks`/`getNotebook`/`getSection`/`getCanvas` filter `deletedAt != null` so the UI still sees it as gone. Same idea one level down: `CanvasController.deleteSelection`/`removeElement`/`_addElementsOp.revert` tombstone (add to `erased`/`deletedObjects`) *and* physically remove from `strokes`/`objects` in the same op — the physical remove is for rendering (painter/hit-test read `page.strokes`/`page.objects` directly, no tombstone filter at that layer), the tombstone is what travels through merge. Page deletion (`CanvasController.deletePage`) explicitly calls `savePage` with `deletedAt` set before the page leaves the in-memory `pages` map (the normal debounced autosave keys off that map and would otherwise never persist the tombstone); `NotebookService.loadPages` prunes any row still referencing a tombstoned page (self-heals a canvas.json structural-LWW race) and guarantees a canvas is never left with zero pages. Soft-deleted notebooks/sections/canvases **and individually-deleted pages** are visible in the **Recycle bin** (Settings → Deleted items → `bin_screen.dart`): `NotebookService.listDeletedItems` reads tombstoned files directly (the normal getters filter them; purged markers are hidden); `restoreBinItem` clears `deletedAt` + bumps `rev` (the restore then *beats the tombstone in LWW on every device* — sync propagates it automatically) and re-links the tree leaf its deletion removed. Restore is disabled while the item's parent is itself deleted (`parentAlive`). **Pages are a 4th bin level** (`BinItemType.page`; `listDeletedItems` scans `pages/*.json` under *alive* canvases only — a deleted canvas's pages are subsumed by the canvas entry): `restorePage` clears the tombstone and re-links the page as a **new row appended at the bottom** of the canvas (original position is unrecoverable after restructuring — same simplification as sections/canvases restoring to end-of-tree). **Open-canvas restore is dispatched** — `NotebookService.restorePage` first calls `SyncService.restorePageInOpenCanvas` (keyed by the `CanvasSyncListener._canvasListeners` registry); if the canvas is open, the live `CanvasController.restorePage` does the row-append + tombstone-clear **in memory + `await flushSaves()`** (a disk-level restore would be clobbered by the controller's own autosave), else the disk path (`getCanvas` → `savePage` clears + bumps → append row → `saveCanvas`) runs. A page's terminal purge (`purgePage`, and the 90-day `_gcPagesUnder` now routes here too — was a local-only delete that leaked the file on Drive) strips content + stamps `CanvasPage.purgedAt` in place; the page file **is** the marker (a page has no subtree), so it needs no folder-delete / `applyPagePurgeLocally` / `isPurgedContentPath` branch — `MergeEngine.mergePage` propagates it (terminal, grow-only, earliest-wins, content stripped). **Super-sections (folders) are a 5th bin level** (`BinItemType.folder`): deleting a super-section moves its whole `FolderNode` subtree into `Notebook.deletedFolders` / `Section.deletedFolders` (`DeletedFolder {node, deletedAt, purgedAt?}` in `tree.dart`) — the contained items' files stay on disk *hidden* (the tree no longer references them, so they don't double-list). `restoreFolder` re-appends the subtree; `purgeFolder` drops the record + purges every contained leaf; synced via the notebook/section LWW (no merge change). Folder delete is confirm-less + animates (`ItemTreeView._deleteFolder`). **Restore-across-devices caveat** (same as the 3 container levels): a restore's row-append can lose to a concurrent higher-rev `canvas.json` (page alive-but-unreferenced there until touched). Restoring a page while its canvas is open leaves the original delete op on that session's undo stack — `deletePage`'s `revert` guards against re-inserting a row for a page that was already re-linked out of band (so undo can't create a duplicate-row corruption); a subsequent redo self-heals on reload. **Purge (stage 2, permanent — bin "Delete forever" / "Empty bin" / the 30-day auto-GC, one mechanism):** upgrades the tombstone to a **terminal `purgedAt` marker** — content subtree hard-deleted from local disk *and* Drive (`DriveService.deleteFolder`, queued persistently in `sync_journal.json`'s `purges` list, drained with retry in `_flushPush` after file uploads so markers land first; idempotent — 404 means another device already did it), while the tiny marker doc survives *forever* (the `notebooks.json` entry / a stripped `section.json`/`canvas.json` — never GC'd; that permanence is what stops a stale offline device's live copy from resurrecting the item). `purgedAt` is **grow-only in merges** (`MergeEngine._withPurgeOverride` — exempt from LWW, earliest wins, forces `deletedAt`), so a restore racing a purge loses deterministically. Purge propagates like any edit: devices pulling a marker run `SyncService._applyPurgeMarkers` → `NotebookService.applyXPurgeLocally` (wipe local content, record the prefix, re-enforce the Drive delete). Push/pull both filter content under purged prefixes via `NotebookService.isPurgedContentPath` + the device-local `purged_index.json` cache (rebuilt each GC walk) — the marker files themselves must keep flowing (they carry the purge), which the filter exempts. Local-only notebooks purge as a plain local delete. Consequence (documented): offline edits to a since-purged item are dropped on reconnect. `NotebookService.runGarbageCollection` (on launch + after every `fullResync`) routes expired tombstones through this same purge path and trims stale page-level tombstone entries. **`SyncService`** orchestrates: `NotebookService._writeAtomic` → `onLocalFileSaved` marks the relPath dirty in a **persistent journal** (`sync_journal.json`, survives kill) → debounced (1.5 s) sequential upload with backoff. Pull = Changes API poll every 15 s: known fileIds are downloaded+merged in place with echo suppression; an **unknown** fileId (a new file from another device) is placed by `DriveService.resolveFileId` (parent-chain walk, folder-cache-accelerated; records the mapping with **no** head so echo suppression can't skip the first download) and pulled inline — only a failed resolve or a 410-Gone falls back to a debounced **`fullResync`** — the one routine used for first-sign-in **bootstrap**, 410 recovery, and the Settings **"Repair sync"** button (`_onSignedIn` only bootstraps when there's no changes token / empty index, else resumes incremental). `fullResync` takes its changes baseline token **before** `listAllFiles` (after = changes pushed during a slow resync are skipped forever), and `_pullFile` skips downloads whose `headRevisionId` matches the last reconciled head — `listAllFiles` must **preserve** previously-reconciled heads instead of stamping current ones, or the resync would skip every file. **Sync-correctness invariants for canvas edits:** every element-mutating op commit must `_stamp` (bump `rev`/`updatedAt`/`deviceId` on) the affected elements or LWW ties and devices never converge; every path that removes an element from a page's arrays must write an `erased`/`deletedObjects` tombstone in the same op (the eraser's old first-apply replay guard skipped exactly this and erased ink resurrected). **Stroke/object deletion is rev-based LWW** (07/12/26 — like the bin's notebook/section/canvas deletion): `EraseTombstone` records the element's `rev` at delete time, and `MergeEngine.mergePage`'s filter keeps an element dead only while `element.rev <= tombstone.rev` (was a binary grow-only `!containsKey`). So **undo/redo never removes a tombstone** (grow-only *storage*; un-tombstoning locally would lose to a device that already pulled it): instead, delete = `_tombstoneFor` (tombstone at `el.rev`), revive = keep the same id and `_bumpAliveOn` (bump rev above the tombstone). rev climbs monotonically across undo↔redo so each action out-revs the last. Same-id revival keeps the **undo stack intact** (earlier ops still find the element by id — the fresh-id approach broke this). Shared helpers `_tombstoneFor`/`_bumpAliveOn`/`_tombstoneSlots`/`_reviveSlots` (+ the `_ElSlot` holder) back every delete path: `deleteSelection`, `removeElement`, `_addElementsOp` (insert/draw/paste/image), `_commitErase` (whole + partial eraser), `deleteLinkedText`, split-paste/`insertTypingContinuation` reverts, and **`_reparentSelection`** (cross-page move = delete-on-source + add-on-target, same ids; the landed copy is bumped above any tombstone on its page, which is what lets a move-back out-rev the stale origin tombstone). **Semantic:** a concurrent higher-rev remote edit resurrects a deleted element (edit-wins-over-delete, same as the bin); a passive copy at/below the tombstone rev stays deleted. Covered by `merge_engine_test` (rev-based deletion group + partial-erase-across-sync), `canvas_controller_test` (undo-chain), `partial_eraser_test`, `split_paste_test`. Writes land on disk via `writeAtomicPublic`/`writeAtomicBytesPublic` (no re-upload loop) and bump `SyncService.dataVersion`, which the home + desktop shells listen to and reload. **Open-canvas live merge:** `CanvasController` registers a `CanvasSyncListener` (keyed by canvas id) in its constructor; `_pullFile` routes pulled page files to `applyRemotePage` (set-union merge into the *same* in-memory `CanvasPage` instance — painter/undo refs stay valid — dropping remotely-tombstoned pages, marking the controller dirty **only when the union holds something Drive lacks**, else it would echo revisions forever) and pulled `canvas.json` to `applyRemoteStructure` (skipped while local structural edits are unflushed). Without this, an open canvas's autosave clobbered every pulled merge with stale in-memory state → the delete-resurrection ping-pong. Perf: `DriveService._folderIds` caches folder path→id (warmed during `listAllFiles`) so uploads don't re-resolve their folder chain with per-segment `files.list` calls. `main.dart`'s `_NoteAppState` (a `WidgetsBindingObserver`) flushes pending on background and syncs on resume. **Gotcha:** the sync-write path must NOT go through `savePage`/`saveNotebook` (those `bumpRev` and re-trigger `onLocalFileSaved` → an upload echo loop) — always use the `*Public` shims. See KNOWN_ISSUES "Cloud sync" for remaining limitations (open-canvas live merge, structural LWW on page add/reorder, no Conflicts Inbox, purge-drops-offline-edits, Google Cloud setup prerequisites).

### Canvas (`lib/canvas/` + `lib/screens/canvas_screen.dart`) — the core

- `canvas_layout.dart`: pure layout — rows stacked vertically, pages left-aligned in rows, `kPageGap`=**0** (pages sit flush for a clean continuous document; a hairline border separates them); produces `PageLayout` rects in canvas space. Inserted PDF pages are normalized to the section's default width (aspect ratio preserved) so the column is uniform-width.
- `canvas_controller.dart` (`CanvasController extends ChangeNotifier`): **the app owns the viewport** (`zoom` + `pan`, exposed as `viewportMatrix`; pdfrx has no camera role). **Smooth scrolling** is ticker-driven: touch drag pans 1:1 (`panImmediate`) then flings on release (`flingBy`, velocity + exp decay); the wheel glides toward a target (`scrollByWheel`, exp smoothing); `stopScrollAnimation()` halts momentum on re-grab. **Scrollbars**: `verticalScrollbarThumb`/`horizontalScrollbarThumb` geometry drawn by the painter; mouse can drag the thumb (`beginScrollbarDrag`/`updateScrollbarDrag`), touch is indicator-only (finger pans directly). Also holds tool state, gesture routing (`startToolGesture/update/end` → page-local, `_activeGestureTool` locks the tool for the gesture), eraser (one undo op per gesture; **Stroke** mode removes whole strokes, **Partial** mode — `eraserPartial`, with adjustable `eraserSize`, both persisted device-local via `SettingsService.setEraserPrefs` — splits a stroke at the erased gap: original tombstoned, surviving point-runs become fresh strokes with the original's style/z, so the erase syncs as tombstone + union like everything else; undo tombstones the segments; a segment created mid-gesture re-splits without tombstoning since it was never persisted), lasso selection (point-in-polygon; live refs + `selectionBounds`), selection drags (move/uniform-resize/rotate, deep-copied before/after; **dropping a moved selection over another page reparents the elements to it** — `_reparentSelection` converts coords to the target's local space, one op over both pages), internal static clipboard, **op-based undo/redo** (`_CanvasOp{apply, revert}` over deep copies, capped 100), page/row structure ops, attachments, over-scroll page-add (right → horizontal page, bottom → append; threshold 96px; suppresses the fling), and **debounced per-page saves** (500ms; `flushSaves` on dispose/export).
- `canvas_painter.dart`: one `CustomPainter(repaint: controller)` — culls to visible pages, paints shadow/bg/pattern or PDF bitmap, elements clipped per page (skipping `editingElementId`), active stroke, lasso, then **screen-space** selection box + handles and over-scroll "+" hints. Stroke outlines cached on the element (`cachedOutline`), built with `perfect_freehand` (highlighter = ×2.6 width, no thinning, 0.35 alpha). **Per-page picture cache** (`page_picture_cache.dart`, `controller.pictureCache`): a page's committed elements are recorded once into a `ui.Picture` (page-local coords, zoom-independent — pictures replay as vectors) and replayed each frame, so pan/zoom cost no longer scales with ink density; zoom-dependent drawing (PDF bitmap, border, pattern, page badge, live stroke/lasso) stays outside the picture. **Invalidation rule (load-bearing):** any code path that visually mutates a page's committed elements must `pictureCache.invalidate(pageId)` or the change won't show until something else invalidates — committed ops ride `_markDirty`; the live eraser (`_eraseAt`), live selection drags (`_updateSelectionDrag`), and remote merges (`applyRemotePage`, which can change visuals *without* reaching `_markDirty`) invalidate explicitly. The editing-skipped element is part of the cache key; a page whose image raster is still decoding re-records each frame until it lands (LRU cap 24 entries, disposed on controller dispose).
- `canvas_screen.dart`: input routing — **stylus + primary-button mouse draw** (fixes the old desktop blocker; S-Pen side button / inverted stylus forces eraser), **touch pans/pinches** (`ScaleGestureRecognizer` limited to touch+trackpad; palm-rejected while pen is down; also no-ops while a scrollbar or text-box grab is active), **wheel scrolls** (Ctrl/Cmd+wheel = zoom at cursor, Shift+wheel = horizontal), **touch double-tap = fit page width**. **Finger drawing** (`SettingsService.fingerDraw`, toggled from the canvas overflow menu, device-local): when on, one finger draws too (`_isDrawingDevice` accepts touch); `_fingerDrawPointer` pins the stroke to the first finger, a **second finger landing cancels the in-progress finger stroke into a pan/pinch** (the scale recognizer was tracking both all along), and the touch double-tap-zoom recognizer is dropped from the gesture map (two quick taps are two dots, not a zoom). Toolbar: 5-tool icon row (pen/highlighter/eraser/lasso/text, `_ToolIconButton`) + a collapsible contextual row below it. App bar: undo, redo, Add sheet, **full screen toggle**, toolbar toggle, overflow (Rename, Export PDF, Pages navigator, Attachments, Page settings, Draw with finger). Add sheet flows: blank page (position picker), horizontal page, insert PDF (with-view + placement, or attachment), image (file picker → asset store), **Take photo** (Android/iOS only, `image_picker` camera → same `_placeImageFromFileBytes` placement as the file-pick flow; uses the system camera intent so no manifest permission needed), paste (internal clipboard, else system text). (No "Text box" entry — the Text tool already places boxes on tap; a redundant sheet item was removed.)
- **Per-canvas viewport memory:** on dispose the controller saves `zoom`/`pan` to `SettingsService.saveCanvasViewport` (device-local `settings.json`, capped at 300, never synced — zoom/pan are screen-specific); the first `setScreenSize` restores it, falling back to fit-first-page. **Per-tool style memory:** `penColor`/`highlighterColor`/`textColor` + `penSize`/`highlighterSize` are separate fields; the old shared `color`/`strokeSize` names survive as getters/setters that route by the active tool, so painter/screen call sites didn't change. **Bookmarks:** `Bookmark {id,name,pageId,createdAt}` lives in `Canvas.bookmarks` (rides canvas.json → synced; carried through `applyRemoteStructure` and both `_rewriteCanvas`/`_rewriteSection` rebuild sites — forget one and moves/copies silently drop bookmarks). UI: overflow → Bookmarks sheet (add-here / jump / delete). Add/remove (`addBookmarkHere`/`removeBookmark`) **flush the canvas then bump `dataVersion`** (`_flushThenReindex`) so a new/removed bookmark is searchable without an app restart — the flush must land before the (debounced) search reindex reads canvas.json, or it reads a stale file. **OS clipboard:** `copySelection` mirrors to the system clipboard via a screen-owned hook (`systemCopyHook` — text-only selections as **HTML + plain text in one clipboard item** (`ClipboardHtml.write` via `htmlFromRuns`, so rich targets keep the styling; plain `Clipboard.setData` only as fallback), a single image's original bytes losslessly, else a PNG cropped from the canvas `RepaintBoundary`); `pasteClipboard` falls back to `systemPasteFallback` (OS image → **OS HTML** → OS plain text) when the internal clipboard is empty. **Rich text paste** (`_pasteRichHtml`): clipboard HTML from browsers/Word/OneNote is parsed by `lib/utils/html_text.dart` (`runsFromHtml`, on the `html` package) into styled `TextRun`s — inline b/i/color/size/family, headings, `<br>`/block breaks, ul/ol as the app's plain-glyph list prefixes, browser-style whitespace collapsing; tables degrade to spaced cells, images/scripts dropped. **Markdown paste** (07/11/26): plain clipboard text that `looksLikeMarkdown` (`lib/utils/markdown_text.dart`; strict detector — ordinary prose never matches) converts via `runsFromMarkdown` into the same styled runs (headings/bold/italic/mono code/lists incl. `- [ ]`→tappable ☐/links/quotes) before landing — one-way Notion-style input conversion; covered by `test/markdown_text_test.dart`. **All text pastes (rich + plain) land via `CanvasController.insertRunsAsText`:** fits on the page → one auto-sized box centered (classic); **taller than the page → split at line boundaries into linked continuation boxes** (`TextElement.linkId`, json `gid`), each on its own **new** page — appended right within the same row when the target row already flows horizontally, else new rows below — one undoable structural op. The splitter (`splitRunsByHeight`, `text_measure.dart`) **measures each candidate chunk standalone** (binary search per chunk) instead of accumulating line metrics — combined vs standalone layouts round line heights differently, so estimation produced boxes a few px over budget; chunk concatenation restores the text exactly. When lasso-selecting any part, **"Cut all parts" / "Delete all parts"** appear — in the **text style row**, since a text-only selection shows that row, never the lasso action row (they're in both; putting them only in the lasso row made them unreachable for text — that was a real bug) (`cutLinkedText`/`deleteLinkedText` — cut merges every part back into ONE clipboard text element, and `pasteClipboard` routes an over-tall single text element back through `insertRunsAsText`, so cut+paste = move the whole flowed text; delete tombstones all parts across pages in one op, continuation *pages* stay). **Typing overflow flows onto the next page too** (07/12/26 — closed the "split applies at paste time only" limitation): `_remeasureEditing` detects the editing box crossing the page bottom and (post-frame, `_handlingOverflow` guard) `_handleTypingOverflow` re-splits via the same `splitRunsByHeight`, truncates the live controller to the fitting chunk, and hands the overflow to `CanvasController.insertTypingContinuation` — which **prepends** to an existing linked continuation on the next page, else drops a linked box on the next page **if it's empty** (blank, no ink/objects, not PDF-backed), else **inserts a fresh page** right after (in-row for horizontal rows, new row below otherwise), one undoable op. If the caret rode the overflow (typing at the end), the session commits and hops to the continuation (`jumpToPage` + a fresh edit session at the mapped caret offset); editing the middle keeps the caret put. Overflow only flows *forward* — no Word-style back-rebalancing on delete (deliberate; "Cut all parts"→paste re-flows manually). Covered by `test/typing_overflow_test.dart`. Image + HTML clipboard I/O goes through `lib/utils/clipboard_images.dart` (`ClipboardImages`, `ClipboardHtml`) on **`super_clipboard`** (works on Android/iOS/macOS/Windows/Linux; replaced `pasteboard`, which was desktop-read-only). Android needs the `SuperClipboardDataProvider` `<provider>` in the manifest (image *copy* hands out content:// URIs) and `minSdk 23` (set in `build.gradle.kts`); `super_native_extensions` downloads its precompiled Rust core at build time — first build after `flutter clean` is slow, not broken. Hooks live on the screen because pixel capture needs the widget tree. **Keyboard shortcuts** (`_onCanvasKey`, a `Focus` around the canvas area; canvas grabs focus on pointer-down): Ctrl/Cmd+C/X/V/D/Z (+Shift=redo)/Y, Delete/Backspace, Esc — all ignored while a text edit is open so the TextField keeps its own keys.
- **Text lists (OneNote-style):** `RichTextController.toggleLinePrefix` adds/removes/cycles plain-glyph line prefixes (`• `, `★ `, `☐ `→`☑ `) on every selected line, and the controller's `value` setter auto-continues the list on Enter (checked `☑` continues as unchecked `☐`; Enter on an *empty* item exits the list). Glyphs are ordinary characters, so painter/PDF export/sync need nothing special. Toolbar buttons appear in the text style row only while editing (`c.isEditingText` → `toggleTextListPrefix`). Covered by `test/rich_text_lists_test.dart`. **Markdown input rules** (07/12/26, `RichTextController._maybeApplyInputRules`, Notion model): typing `# `/`## `/`### ` at a line start sets a heading typing style (marker consumed; Enter resets to base — see `_baseDefaults`), `- `/`* `→`• `, `[ ] `/`[x] `→`☐ `/`☑ ` (also swallows an existing list glyph), `> `→`│ `+italic, and completing `**bold**`/`*italic*`/`` `code` `` restyles the content and drops the markers (single `_` deliberately inert — snake_case). Backspace immediately after a rule restores the raw text and typing style (`_maybeRevertRule`); rules skip mid-IME composition. **Gotcha:** rules that change the typing style set `_suppressStyleAdopt`, consumed by `_onEditingChanged`'s caret-move adoption — without it the rule's own caret move re-adopts the surrounding style and clobbers the heading (same mechanism as the style-while-typing fix). Known quirk: a heading rule fired on an *empty* line has no characters to backspace, so its revert is unreachable there (escape via toolbar size). Covered by `test/rich_input_rules_test.dart`. **Checkboxes toggle by tapping them on the canvas** (any tool, no editor opens — mirrors link taps): `checkboxOffsetAt` (`text_measure.dart`, pure, unit-tested) hit-tests the tap against the line-leading ☐/☑ glyph's drawn box (padded; indented/nested items work; a mid-sentence ☐ is not a checkbox) → `CanvasController.toggleCheckboxAt` flips the character as one undoable, `_stamp`ed op; routing lives beside the `_urlAt` checks in `canvas_screen`'s pointer-up (both tap sites — general + lasso). Covered by `test/checkbox_toggle_test.dart`.
- **Text-edit gotcha (fixed):** the style row is wrapped in `TextFieldTapRegion` — without it, tapping any style button counted as "outside" the editing `TextField`, firing `onTapOutside` → committing the session and collapsing the selection, so styling a selected range was impossible. Any new control that must not dismiss the text editor belongs inside that region. **Second gotcha (fixed 07/08/26):** `_onPointerDown`'s early return for taps inside the editor rect skips setting `_downPosition`, so the matching `_onPointerUp` computed `moved == false` and ran the tap-handling branch → `_handleTextTap` hit the very box being edited → `_startTextEdit` re-ran with a **fresh controller**, wiping the in-box selection the instant the pointer lifted ("selection dies on lift", both platforms). Fix: `_pointerInTextEditor` flag set on that early return; the pointer-up returns immediately when set (the TextField already owned the whole gesture). Any new pointer-up handling must respect that flag.
- **Tool options are gated behind a re-tap, in both normal and full-screen layouts.** `CanvasController.setTool(t)` is the single source of truth: tapping the *already-active* tool toggles `toolOptionsOpen`; tapping a *different* tool switches and always closes it. `_buildToolContextRow` (top-level function, shared by `_ToolOptionsPanel` and full screen's floating control) returns `null` — meaning render nothing — unless `toolOptionsOpen` is true, **except** for panels reflecting something already in progress (actively editing text, a text box selected via lasso, or an active lasso selection/clipboard content) which stay visible regardless, since hiding those would hide the very thing the user is doing. **The options panel must never move the canvas** (load-bearing, was a complaint): in the normal layout `_CanvasToolbar` renders **only the fixed tool row**, and the options live in a separate `_ToolOptionsPanel` **overlaid** at the top of the canvas (a `Positioned` in the canvas `Stack`), animating open/closed via `AnimatedSwitcher`+`SizeTransition` — so the `Expanded` canvas area's constraints (and `setScreenSize`) never change when options toggle. The panel is wrapped in an **opaque `Listener`** so taps on it (a swatch, or a gap) don't fall through to the canvas `Listener` behind and close it mid-interaction. The first tap outside an open options panel or the full-screen tool picker dismisses it without also starting a draw/pan gesture (`canvas_screen.dart`'s `_onPointerDown` intercepts this before anything else). **Full screen** (`_isFullScreen`, toggled by the app-bar button) hides the app bar and toolbar entirely, replaced by two floating controls: an exit button, and a tool control that starts collapsed to the active tool's icon, expands to the 5-tool picker on tap, and shows the same options panel as normal mode when the active tool is tapped again within the picker.
- **Text** is **rich (per-run styling)**: a `TextElement` holds a list of styled `TextRun`s (`text` is their concatenation); `text_measure.dart` builds a multi-style span (`textSpanForElement`) for painting and auto-sizing. Boxes **auto-size** to content (`autoTextRect`, re-measured live each keystroke), wrapping only at the page's right edge — never a fixed width. Editing uses a `RichTextController` (`rich_text_controller.dart`) that keeps a per-character style array (reconciled on each edit via prefix/suffix diff), renders it through `buildTextSpan`, and `applyToSelection` mutates only the selected range **without changing the selection** (so repeated size nudges work); a collapsed caret sets the typing style. `CanvasController` style setters route to `_richController.applyToSelection` while editing, else to the whole selected box(es) (`updateSelectedText`), else to the defaults; `align` is whole-box. On commit `runsFromController` collapses styles back into merged runs. In the text tool, **tap a box to edit, drag to move** (reuses the selection move/reparent pipeline); lasso mode edits text on tap. Text ignores box-resize (`scaleBy` no-op for size); text-only selections have no resize corners. Editing UI is a positioned `TextField(controller: RichTextController)` overlay committing on tap-outside; the painter skips `editingElementId`. **PDF export is per-run:** `placedRunFragments` (`text_measure.dart`) lays the box out with the *same* `TextPainter` the painter uses and splits it into per-line styled fragments with positions (mixed sizes baseline-align via each fragment's own selection-box top); `pdf_exporter.dart`'s `_drawText` draws each fragment with its own `PdfStandardFont`/color at that position (+0.15 em top nudge to match the painter's 1.3 line height), so line breaks, alignment, and styling match the screen exactly.

### PDF export (`lib/services/pdf_exporter.dart`)

`PdfExporter` interface; `SyncfusionPdfExporter` implements spec §10: one Row → one PDF page; multi-page rows merge **flush** into one wide page sized exactly to the summed page widths (see Gotchas — page size must be assigned via a fresh `PdfPageSettings`, never by mutating `.size`); PDF-backed pages draw the original imported page via `createTemplate()`/`drawPdfTemplate` (**vector — text stays selectable**); strokes exported as filled vector paths using the *same* perfect_freehand options as the painter; text as real PDF text (standard fonts Helvetica/Times/Courier); images embedded; highlighter via `setTransparency(0.35)`. Background **pattern** (ruled/grid/dotted) is drawn as vector lines/dots on blank pages (`_drawPattern`, mirroring `CanvasPainter._paintPattern`'s exact 26pt-spacing geometry) — PDF-backed pages skip it since the imported page image covers the whole page, same as on screen. Saved via `FilePicker.saveFile` (desktop returns a path without writing — the screen writes bytes if needed).

### Branding (`assets/branding/`)

App icon is the "Stroke Page" mark — a rounded page traced in one open, tapered stroke (accent color, no fill), chosen from three mockup directions (`CANVAS_SPEC.md`-adjacent design work, not otherwise tracked in a file). Source PNGs (`icon_flat.png`, `icon_foreground_light.png`, `icon_foreground_dark.png`) were rasterized at 1024×1024 directly via `dart:ui` (`PictureRecorder`/`Canvas`/`Picture.toImage`) in a temporary throwaway test script — no external image tool needed; delete-after-use if regenerating. `flutter_launcher_icons` (config lives in `pubspec.yaml`) generates the real Android adaptive icon (`icon_flat.png` flattened fallback, white `adaptive_icon_background`, `icon_foreground_light.png` foreground), Windows `.ico`, and macOS `.icns`-equivalent set — regenerate with `dart run flutter_launcher_icons` after changing the source images or config. iOS/web are intentionally left off (not project priorities).

**Splash screen is hand-rolled, not package-generated.** `flutter_native_splash` cannot currently be added to this project — every version requires either an `xml` range incompatible with `syncfusion_flutter_pdf ^33.2.15`'s `xml ^7.0.1`, or a `meta` version newer than what the installed Flutter SDK bundles; verified via `flutter pub add --dry-run`, not assumed. Android's native splash is instead wired by hand: `values/colors.xml` + `values-night/colors.xml` define `splash_background` (matching `AppTheme`'s light/dark canvas colors exactly, so there's no color flash handing off to the first Flutter frame); `drawable/launch_background.xml` + `drawable-v21/launch_background.xml` (pre-Android-12 path) are identical layer-lists referencing `@color/splash_background` + `@drawable/splash_logo`; `drawable-xxxhdpi/splash_logo.png` + `drawable-night-xxxhdpi/splash_logo.png` are the transparent-background mark (light/dark accent) — **must** live in a density-qualified bucket (not the unqualified `drawable/`), since an unqualified bucket is treated as baseline density and a 1024px source image would render 3–4× too large on real devices; `values-v31/styles.xml` redeclares `LaunchTheme` with the Android 12+ `windowSplashScreenBackground`/`windowSplashScreenAnimatedIcon`/`windowSplashScreenIconBackgroundColor` items (a single file — no `values-night-v31` needed, since the `@color`/`@drawable` references it points at already resolve their own night variants). Windows/macOS/Linux native splash screens are out of scope — no equivalent was hand-implemented; see `KNOWN_ISSUES.md`.

### Gotchas

- Elements/Section/pages are mutable by design; **undo correctness depends on ops snapshotting `deepCopy()`s** — never store live element refs in an op.
- `CanvasController.setScreenSize` is called during build → it must never `notifyListeners()` synchronously.
- Stroke colors are absolute; canvas page backgrounds are user-chosen (default white), so theme changes don't affect page/ink contrast anymore.
- On macOS, `com.apple.security.app-sandbox` is enabled but no file-access entitlement is granted yet — file picking may fail in release builds (see KNOWN_ISSUES).
- **Never mutate `sf.PdfPageSettings.size` (or `.orientation`) on an existing settings object in `pdf_exporter.dart`.** The setter re-derives width/height as `(min, max)` of the *current* orientation (default `portrait`), silently swapping them whenever width > height — which is every horizontally-merged row and every landscape-shaped PDF page. Always assign a **fresh** `sf.PdfPageSettings(Size(w, h))` (size-only constructor arg — no orientation) to `section.pageSettings` instead; that stores the size verbatim. Covered by `test/pdf_export_test.dart` — extend those tests if the exporter's page-sizing code changes.
